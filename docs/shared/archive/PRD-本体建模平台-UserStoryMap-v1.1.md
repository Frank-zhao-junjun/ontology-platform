## 1. Ralph Loop 方法论

### 1.1 什么是 Ralph Loop

Ralph Loop 是 Jeff Patton 在 User Story Mapping 中引入的迭代产物发现循环。名字来源于《布偶秀》里只会问"为什么"的 Ralph——而"为什么"恰好是整个循环的灵魂。

```
        ┌─────────────────────┐
        │   Who（为目标用户）   │
        └─────────┬───────────┘
                  │
                  ▼
        ┌─────────────────────┐
        │  What（他们想达成什么）│←──────────┐
        └─────────┬───────────┘            │
                  │                        │
                  ▼                        │
        ┌─────────────────────┐            │
        │  Why（为什么这很重要）│────────────┘
        └─────────────────────┘    （每轮迭代回到 What，打磨粒度）
```

### 1.2 三圈打磨规则

每个 User Story 在本 PRD 中至少经过 **三轮 Ralph Loop**：

| 轮次 | 焦点 | 产出 |
|------|------|------|
| **R1（成立性）** | Who 是否真实？What 是否可验证？Why 是否触及业务指标？ | 故事一句话 |
| **R2（可验收）** | 加入 Acceptance Criteria（GIVEN/WHEN/THEN） | 验收标准 |
| **R3（约束与边界）** | 这个故事的完成不应依赖什么？必须兼容什么？ | 约束条件 |

### 1.3 故事格式约定

```
US-XXX [层] [优先级] — 故事标题
├─ Who： 具体角色（来自 Persona 列表）
├─ What：一句话描述 TA 想达成的结果（不是功能，是状态变化）
├─ Why：  底层动机（可追溯到业务指标或痛点指标）
├─ AC：   GIVEN/WHEN/THEN 验收标准
└─ 约束： 依赖项 / 非依赖项 / 兼容性要求
```

### 2.2 详细画像

#### Persona A：张工 — 业务领域专家（SME）

- **年龄/背景**：45 岁，20 年制造业经验，历任车间主任→生产部长→精益顾问
- **技术水平**：会用 ERP（SAP），不会写代码，不会 SQL。能画业务流程图
- **日常工作**：制定生产策略、审核工单、协调物料、处理异常
- **与平台的关系**：他是**知识的提供者**——他定义"什么是物料齐套""什么时候工单能下达"。他的知识是本体模型的原材料
- **关键指标关注**：准时完工率、工单平均完成时间、物料周转天数
- **最大痛点**：他的业务知识全部在脑子里和 Excel 里。他退休/调岗的那一天，这些规则就丢了

#### Persona B：李工 — 本体建模工程师（Ontology Engineer）

- **年龄/背景**：32 岁，计算机专业，5 年 DDD 实践，做过微服务架构
- **技术水平**：熟练 DDD 战术模式（聚合/实体/值对象/领域事件），会写 SQL/API 集成代码
- **日常工作**：需求分析→领域建模→设计聚合→定义接口→Code Review
- **与平台的关系**：他是平台的**核心用户**——他把张工的业务知识转化为结构化的本体模型。建模质量直接决定 Agent 能多安全地操作业务系统
- **最大痛点**：每次新项目从零开始画 ER 图、写接口文档。之前项目定义的"生产订单"模型无法复用，因为散落在代码、Wiki、飞书文档里

#### Persona C：王工 — AI 应用开发工程师（AI Developer）

- **年龄/背景**：28 岁，全栈+AI，2 年 LLM 应用开发经验
- **技术水平**：Python/Node.js、LangChain/LlamaIndex、MCP 协议、Prompt Engineering
- **日常工作**：开发企业内部 AI 助手（如"生产计划助手""物料查询助手"）
- **与平台的关系**：他是本体的**消费者**——他通过 Manifest 和 MCP 协议让 AI Agent 安全地操作业务系统
- **最大痛点**：Agent 幻觉导致它调了不该调的 API（比如把未审核的工单直接下达了）。他不知道 Agent "应该能做什么、不能做什么"——因为没有一份机器可读的"权限清单"

#### Persona D：赵工 — 平台管理员（Platform Admin）

- **年龄/背景**：35 岁，运维+安全背景，负责企业 IT 基础设施
- **技术水平**：Linux/容器化、IAM、安全审计、合规
- **日常工作**：用户管理、权限分配、系统监控、安全审计
- **与平台的关系**：他是平台的**守门人**——他管理谁能访问什么、Agent 能在哪个沙箱运行、数据源连接是否安全
- **最大痛点**：权限配置散落在多个系统（ERP 的、数据库的、API 网关的），出事时审计链路断裂——不知道是"人改的"还是"Agent 改的"

#### Persona E：陈工 — 运营分析师（Operations Analyst）

- **年龄/背景**：30 岁，工业工程专业，擅长数据分析
- **技术水平**：熟练 BI 工具（Power BI/Tableau）、会写基础 SQL
- **日常工作**：制作生产报表、分析 KPI 趋势、识别瓶颈
- **与平台的关系**：他是**指标和事件的消费者**——他用本体定义的指标口径和事件数据来确保"所有人看的是同一个数字"
- **最大痛点**：ERP 系统报的"准时完工率"是 78%，MES 报的是 82%，BI 报表又算出来 75%。三个系统三个数，他不知道信哪个——因为没人说清楚"准时"到底怎么算

## 3. 用户旅程主干（Backbone）

Backbone 是用户从"我有一个业务场景"到"AI Agent 安全运行在生产环境"的完整旅程。

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  ① 领域定义   │───→│  ② 语义建模   │───→│  ③ 行为建模   │───→│  ④ 事件建模   │───→│  ⑤ 治理配置   │
│  (张工主导)   │    │  (李工主导)   │    │  (李工主导)   │    │  (李工主导)   │    │  (赵工主导)   │
│  划定边界      │    │  结构化知识    │    │  定义规则     │    │  编排联动     │    │  权限与安全   │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
                                                                                      │
                                                                                      ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ ⑩ 运营分析   │←───│  ⑨ 事件消费   │←───│  ⑧ Agent运行  │←───│  ⑦ MCP消费    │←───│  ⑥ 制品发布   │
│  (陈工主导)   │    │  (陈工主导)   │    │  (王工主导)   │    │  (王工主导)   │    │  (李工主导)   │
│  指标与洞察   │    │  联动与追溯   │    │  安全执行     │    │  集成开发     │    │  版本化输出   │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

### 3.1 十步简介

> **注**：建模工作流（US-G05：草稿→审核→发布）是横跨步骤 ②–⑥ 的治理流程，确保模型变更经过双人复核后才能编译为 Manifest。

| 步骤 | 主干名 | 主导角色 | 一句话 |
|------|--------|---------|--------|
| ① | 领域定义 | 张工(SME) | 划定限界上下文和业务场景的边界 |
| ② | 语义建模 | 李工(OE) | 将业务概念转化为聚合根/对象/属性/关系 |
| ③ | 行为建模 | 李工(OE) | 定义行为、校验规则、指标公式、事务边界 |
| ④ | 事件建模 | 李工(OE) | 定义领域事件、集成事件、事件路由和处理器 |
| ⑤ | 治理配置 | 赵工(Admin) | 配置角色权限、字段级权限、AI Agent 沙箱策略 |
| ⑥ | 制品发布 | 李工(OE) | 编译本体为版本化 Agent Manifest |
| ⑦ | MCP 消费 | 王工(AID) | Agent 通过 MCP 协议查询本体、获取工具白名单 |
| ⑧ | Agent 运行 | 王工(AID) | Agent 在执行时受三通道防御保护 |
| ⑨ | 事件消费 | 陈工(OA) | 领域事件触发跨系统联动、更新指标 |
| ⑩ | 运营分析 | 陈工(OA) | 基于统一指标口径分析生产运营表现 |

## 4. User Stories — 语义层

> 语义层覆盖主干步骤 ①②：领域定义 + 语义建模

### 4.1 限界上下文管理

---

#### US-S01 [语义层] [P0-NOW] — 创建限界上下文并绑定本体

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 在平台上创建一个名为"生产制造"的限界上下文，它与一个本体 1:1 绑定。创建完成后，这个上下文成为后续所有建模操作的命名空间 |
| **Why** | 限界上下文是 DDD 战略设计的核心——不先划定边界，后续"物料""生产订单"这些概念属于哪个领域都不清楚。1:1 绑定本体意味着：**一个上下文 = 一个独立可发布的 Agent Manifest** |
| **AC-1** | GIVEN 用户进入平台首页 WHEN 点击"新建限界上下文" THEN 弹出表单，要求填写：上下文名称、描述、所属领域标签 |
| **AC-2** | GIVEN 已填写上下文名称"生产制造" WHEN 点击"创建" THEN 平台自动创建一个同名本体并绑定，显示在上下文列表中 |
| **AC-3** | GIVEN 已存在名为"生产制造"的上下文 WHEN 尝试创建同名上下文 THEN 平台提示"该限界上下文已存在"，不允许重复 |
| **AC-4** | GIVEN 上下文已创建 WHEN 查看上下文详情 THEN 可看到：绑定的本体 ID、创建时间、包含的聚合根数量（初始为 0） |
| **约束** | 限界上下文名称在平台内全局唯一；删除上下文需要先清空其下的所有模型元素 |

#### US-S02 [语义层] [P1-NEXT] — 在上下文中定义业务场景

| 维度 | 内容 |
|------|------|
| **Who** | 张工(SME) — 业务领域专家 |
| **What** | 在"生产制造"限界上下文下，定义两个业务场景："面向库存生产(MTS)"和"面向订单生产(MTO)"。每个场景可以有不同的行为规则和校验逻辑 |
| **Why** | 同一个"生产订单"在不同业务场景下行为不同——MTS 允许自动补货触发，MTO 必须严格按销售订单驱动。场景是行为规则和事件处理器的分类维度 |
| **AC-1** | GIVEN 进入"生产制造"上下文 WHEN 点击"新建业务场景" THEN 创建"面向库存生产"和"面向订单生产"两个场景 |
| **AC-2** | GIVEN 场景已创建 WHEN 在语义建模中选择聚合根 THEN 可指定该聚合根适用于哪些场景（支持多选） |
| **AC-3** | GIVEN 后续定义行为规则 WHEN 关联规则到场景 THEN 规则仅在指定场景下生效 |
| **AC-4** | GIVEN 后续定义事件处理器 WHEN 关联处理器到场景 THEN 处理器仅在指定场景下激活（见 US-E04） |

### 4.2 聚合根与对象建模

---

#### US-S03 [语义层] [P0-NOW] — 定义聚合根

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 在"生产制造"上下文中定义四个聚合根：生产订单、物料、BOM、工艺路线。每个聚合根是其他对象（实体、值对象）的入口点 |
| **Why** | 聚合根是 DDD 事务一致性的边界，也是 AI Agent 访问业务数据的唯一入口——**Agent 不能绕过聚合根直接操作子对象**。这是 AI 安全的第一道防线 |
| **AC-1** | GIVEN 进入上下文建模界面 WHEN 点击"添加聚合根" THEN 弹出表单：名称(必填)、标识符(中文名+英文code)、描述、所属场景 |
| **AC-2** | GIVEN 已定义"生产订单"聚合根 WHEN 添加子对象（如"工序"） THEN 子对象自动挂载在该聚合根下，不允许独立存在 |
| **AC-3** | GIVEN 定义了多个聚合根 WHEN 后续定义关系类型 THEN 允许跨聚合根建立关系（如"生产订单→物料"） |
| **AC-4** | GIVEN 聚合根已创建 WHEN 查看数据血缘图 THEN 可看到该聚合根依赖的数据源和获取方式 |
| **约束** | 聚合根数量下限：一个限界上下文至少 1 个；上限：无硬限制，但建议单个上下文 ≤ 10 个聚合根以保证可维护性 |

#### US-S04 [语义层] [P0-NOW] — 定义对象类型与属性

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 为"生产订单"聚合根定义对象类型（如"生产订单头""工序""物料组件"），每个对象类型包含属性定义（字段名、类型、必填/可选、枚举值）。详见附录 C 制造域参考模型 |

| **AC-5** | GIVEN 对象已定义 WHEN 标记为"接口抽象" THEN 该对象只声明属性签名不包含实现，供其他对象继承/实现 |

#### US-S06 [语义层] [P1-NEXT] — 定义对象状态与状态机

| 维度 | 内容 |
|------|------|
| **Who** | 张工(SME) + 李工(OE) — 业务专家 + 建模师 |
| **What** | 为"生产订单"定义其生命周期状态：创建→已下达→执行中→已报工→已产出→技术关闭。每个状态转换由**行为驱动**（Behavior 是状态转换的唯一原因），状态转换完成后**发布领域事件**（见 US-E01），事件进而触发**事件处理器**（见 US-E04）。完整因果链：**行为执行 → 状态转换 → 领域事件发布 → 事件处理器执行** |
| **Why** | 状态机是业务规则的结构化表达——"工单在'创建'状态下不能报工"这个规则不是写在 if-else 里，而是定义在模型中，Agent 执行前校验层就能拦截。状态机也是事件处理器矩阵的关键维度（见 US-E04）。明确行为→状态→事件的因果链，避免与 E04 的"事件触发行为"产生概念混淆——S06 的"触发行为"指的是"哪个行为驱动了状态转换"（前因），E04 的"处理行为"指的是"事件发生后执行什么行为"（后果），二者是同一因果链上的不同环节 |
| **AC-1** | GIVEN 选中"生产订单"对象 WHEN 打开"状态定义"面板 THEN 可添加状态节点：状态名、状态码、是否为终态、是否为初态 |
| **AC-2** | GIVEN 已定义 6 个状态 WHEN 定义状态转换 THEN 选择"从状态→到状态"，关联**驱动行为**（如"创建→已下达"由行为"生产订单下达"驱动执行），并自动关联该行为声明的领域事件（见 US-E01 AC-3） |
| **AC-3** | GIVEN 状态机已配置 WHEN 校验层检查 Agent 输出 THEN 非法状态跳转（如"已关闭→执行中"）被拒绝 |
| **AC-4** | GIVEN 状态机已配置 WHEN 查看状态转换图 THEN 可视化显示为有向图，初态为绿色、终态为灰色、当前活跃转换高亮；边上标注驱动行为名称和发布的事件名称 |
| **约束** | 状态转换必须通过行为驱动——不允许跳过行为直接修改状态。一个状态转换关联一个驱动行为；驱动行为执行成功后自动完成状态转换并发布事件（原子操作） |

#### US-B01 [行为层] [P0-NOW] — 定义行为

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 定义 6 个核心行为：创建生产订单、生产订单下达、物料领用、生产报工、成品产出、工单技术关闭。每个行为包含：名称、关联聚合根、关联业务场景、**调用模式**（主动调用 / 事件驱动 / 两者皆可）、输入参数、前置校验规则、声明式事件发布列表 |
| **Why** | 行为是本体的核心执行元语——Agent 不是"直接操作数据库"，而是"调用行为"。行为自带校验规则和事件发布声明，Agent 不需要（也不应该）知道校验逻辑怎么写。同一个行为可在两种模式下被触发：①**主动调用模式**——Agent 或用户直接调用（如 B01 场景）；②**事件驱动模式**——由事件处理器根据矩阵规则自动触发（如 E04 场景，处理器引用的"处理行为"即来自本行为库）。无论哪种模式，行为内部的校验规则、事务边界、事件发布逻辑完全一致——模式只影响"谁触发了行为"，不影响"行为做什么" |
| **AC-1** | GIVEN 进入行为管理界面 WHEN 点击"新建行为" THEN 填写：行为名称(中英文)、关联聚合根、关联业务场景、**调用模式**(主动调用/事件驱动/两者皆可，默认"两者皆可")、描述 |
| **AC-2** | GIVEN 行为"生产订单下达"已创建 WHEN 定义输入参数 THEN 至少包含：生产订单 ID(必填)、下达日期(可选)、备注(可选) |
| **AC-3** | GIVEN 行为已定义 WHEN 关联校验规则 THEN 可从已定义的规则库中选择（如"物料齐套校验"），并指定规则执行顺序 |
| **AC-4** | GIVEN 行为已定义 WHEN 配置事件发布 THEN 声明执行成功后发布哪些领域事件（如"生产订单已下达"） |
| **约束** | 行为必须通过聚合根入口——行为"物料领用"不能直接操作"物料"聚合根内部，必须通过聚合根暴露的方法。标记为"仅事件驱动"的行为不出现在 MCP 工具白名单中（Agent 不能直接调用，只能由事件处理器触发） |

### 5.2 校验规则

---

#### US-B03 [行为层] [P0-NOW] — 定义校验规则并关联行为

| 维度 | 内容 |

### 5.4 事务边界

---

#### US-B06 [行为层] [P1-NEXT] — 定义事务边界

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 为"下达工单"行为定义事务边界。事务边界规定了哪些数据变更必须在同一个原子操作中一起成功或一起回滚，不允许中间态存在。具体操作：①更新工单状态"创建"→"已下达" ②扣减物料预留量（按 BOM 展开）③生成工序任务（按工艺路线展开）——这三个操作必须原子执行 |
| **Why** | 事务边界防止数据中间态——如果工单状态变成"已下达"但物料没扣减，就会出现"已下达但料不够"的脏数据。Agent 不需要理解事务，但 Manifest 需要告诉执行引擎"这些操作必须一起成功或一起回滚" |
| **AC-1** | GIVEN 选中行为 WHEN 进入"事务边界"配置 THEN 可添加"关联事务"：选择该行为执行时必须原子化的关联操作，每个操作描述为"对象+属性+变更" |
| **AC-2** | GIVEN 事务边界已定义 WHEN 查看事务清单 THEN 列出：①生产订单.状态=已下达 ②物料.预留量-=需求量 ③工序.状态=待执行 |
| **AC-3** | GIVEN 事务边界已定义 WHEN Manifest 编译 THEN 事务边界信息写入 Manifest 的 transaction_boundary 字段 |
| **AC-4** | GIVEN 执行引擎执行行为 WHEN 事务中任一操作失败 THEN 整个事务回滚，所有已执行的变更撤销 |

### 5.5 副作用配置

---

#### US-B07 [行为层] [P1-NEXT] — 配置行为副作用

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 为行为配置副作用（Side Effect）声明——行为执行后，除了发布领域事件外，还需要触发的非核心操作。如"生产订单下达"的副作用：发送通知给车间主任、更新 ERP 中的工单状态同步标记 |
| **Why** | 副作用与领域事件不同——事件是"发生了一个事实"（语义层面），副作用是"还需要做一件附带的事"（操作层面）。区分两者让 Manifest 消费方能区分"核心流程"和"附加操作"，核心流程失败需回滚，副作用失败可以异步重试 |
| **AC-1** | GIVEN 选中行为 WHEN 进入"副作用"配置 THEN 可添加副作用：类型(通知/同步/日志)、目标(人/系统/队列)、内容模板 |
| **AC-2** | GIVEN 副作用已配置 WHEN 行为执行成功 THEN 副作用异步触发，不阻塞主流程 |
| **AC-3** | GIVEN 副作用执行失败 WHEN 查看副作用执行日志 THEN 标记为"失败可重试"，不影响主事务结果 |
| **约束** | 副作用必须声明为"最终一致性"——不能将副作用纳入事务边界。事务边界内只放核心操作 |

### 6.2 集成事件

### 6.3 事件存储

---

#### US-E02 [事件层] [P1-NEXT] — 配置事件存储

| 维度 | 内容 |
|------|------|
| **Who** | 赵工(Admin) + 李工(OE) |
| **What** | 配置事件存储后端（如 Kafka Topic 或 PostgreSQL Event Store），所有领域事件持久化写入该存储 |
| **Why** | 事件存储是审计追溯和事件溯源的基础——"谁在什么时间下达了哪个工单"应该能从事件日志中完整还原。**与 E03/E04 的关系**：E03(事件路由)和 E04(事件处理器矩阵)是**建模期定义**——定义事件的路由规则和处理逻辑，不依赖事件存储即可完成建模；E02(事件存储)是**运行期基础设施**——让事件持久化以供审计和溯源。MVP 阶段可使用**进程内事件通道**（同步内存分发）验证 E03/E04 的路由和处理逻辑，待 E02 配置完成后切换为持久化事件总线 |
| **AC-1** | GIVEN 进入事件存储配置 WHEN 选择存储类型(Kafka/PG/Redis Stream) THEN 填写连接信息和 Topic/Table 映射 |
| **AC-2** | GIVEN 事件存储已配置 WHEN 行为发布事件 THEN 事件 Payload 自动写入事件存储 |
| **AC-3** | GIVEN 事件已存储 WHEN 查询某个聚合根的事件历史 THEN 按时间顺序返回该聚合根的全部事件序列（事件溯源） |
| **约束** | 未配置事件存储时，平台使用进程内事件通道——路由和处理逻辑正常执行但不持久化，重启后事件丢失。生产环境必须配置事件存储 |

### 6.4 事件路由

#### US-E03 [事件层] [P0-NOW] — 定义事件路由

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 为每个领域/集成事件定义路由规则——事件应被分发到哪些限界上下文和外部系统。例如："生产订单已下达"→ 路由到"物料管理"上下文 + "MES"外部系统 |
| **Why** | 事件路由是"事件会影响哪些领域+系统"的显式声明——没有路由规则，事件发出去没人知道该往哪送。路由规则是事件驱动架构的"接线图" |
| **AC-1** | GIVEN 事件"生产订单已下达"已存在 WHEN 配置路由 THEN 可添加多个路由目标：限界上下文(如"物料管理")、外部系统(如"MES") |

### 6.5 事件处理器矩阵

---

#### US-E04 [事件层] [P0-NOW] — 定义事件处理器（行为×场景×状态矩阵）

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师，张工(SME) 确认业务规则 |
| **What** | 为每个领域事件定义事件处理器——即"当事件 X 发生时，在场景 Y 下、对象处于状态 Z 时，应执行行为 W"。这形成一个三维矩阵：**事件 × 业务场景 × 对象状态 → 处理行为**。处理行为 W 复用 US-B01 中定义的行为库——同一个行为既可被 Agent 主动调用（主动调用模式），也可被事件处理器触发（事件驱动模式），行为内部的校验规则和事件发布逻辑完全一致 |
| **Why** | 这是你提出的关键需求——"事件处理器：行为#1(业务场景+行为+状态)"。没有场景和状态的约束，事件处理器会变成"所有事件都触发所有行为"的混乱。矩阵确保"生产订单已下达"事件在 MTS 场景下触发"自动补料"，在 MTO 场景下触发"按单领料"——同一个事件，不同的上下文，不同的行为。**与 S06 的关系**：S06 定义状态机的"驱动行为"（前因：行为→状态转换→事件），E04 定义事件的"处理行为"（后果：事件→矩阵匹配→处理行为），二者构成完整的因果链（详见 S06 What 字段） |
| **AC-1** | GIVEN 事件"生产订单已下达"已定义 WHEN 添加处理器 THEN 配置：目标行为(从 US-B01 行为库选择，仅可选调用模式为"事件驱动"或"两者皆可"的行为)、适用场景(从场景列表选择)、对象前置状态(从状态机选择) |
| **AC-2** | GIVEN 6 个事件 × 2 个场景 × 6 个状态 WHEN 查看事件处理器矩阵 THEN 以表格形式展示所有处理器映射（详见附录 E） |
| **AC-3** | GIVEN 事件到达 WHEN 匹配处理器 THEN 按优先级匹配：先匹配"场景+状态"完全符合的处理器；若无精确匹配，尝试"场景匹配+状态任意"；若无匹配则忽略 |
| **AC-4** | GIVEN 处理器执行 WHEN 对象状态不满足前置条件 THEN 处理器跳过执行并记录日志 |
| **约束** | 一个事件在同一个(场景,状态)组合下最多关联一个处理器——避免处理器冲突。处理行为必须是已在 B01 行为库中定义的行为，不支持内联定义 |

### 7.2 AI Agent 沙箱

---

#### US-G04 [治理层] [P0-NOW] — 配置 AI Agent 沙箱

| 维度 | 内容 |
|------|------|
| **Who** | 赵工(Admin) — 平台管理员 |
| **What** | 为 AI Agent 配置沙箱执行环境：Agent 默认无任何权限，必须在沙箱配置中显式授权它调用哪些 MCP 工具、访问哪些聚合根、执行哪些行为。沙箱还有一个"最大操作频率"限制（防止 Agent 意外循环调用）。最小权限原则：Agent 默认无权限，需显式授权 |
| **Why** | 沙箱的权限白名单**来自本体模型**。本体定义了"Agent 能做什么"，沙箱是执行这个定义的容器。Agent 在沙箱里看不到数据库、看不到 API 密钥、没有文件系统——它只能调用你显式授权的那几个 MCP 工具。详见附录 B |

| **AC-3** | GIVEN 沙箱已配置频率限制(如 10次/秒) WHEN Agent 超过频率 THEN 沙箱返回 429 限流错误 |
| **AC-4** | GIVEN 沙箱中的 Agent 执行操作 WHEN 查看审计日志 THEN 可追溯到：哪个 Agent、哪个沙箱、哪个行为、什么时间、操作了什么数据 |
| **约束** | 沙箱是最小权限原则的物理实现——不是"默认允许，黑名单拦截"，而是"默认拒绝，白名单放行"。沙箱与本体模型的关系见附录 B |

### 7.3 建模工作流

---

#### US-G05 [治理层] [P0-NOW] — 建模工作流与审批（草稿→审核→发布）

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) 提交模型变更，张工(SME) 审核业务语义正确性，赵工(Admin) 最终批准发布 |
| **What** | 本体模型的变更遵循三阶段工作流：**草稿(Draft)** → **审核中(Review)** → **已发布(Published)**。处于草稿状态的模型可自由编辑；提交审核后模型进入只读锁定状态，审核人可在模型上添加批注和建议；审核通过后由管理员批准发布，此时模型生成新的 Manifest 版本（见 US-A01）。任何阶段都可查看完整的变更历史和审批记录 |
| **Why** | 本体模型是 Agent 行为的"宪法"——模型中的一个错误（如把"下达工单"的校验规则写错）可能导致 Agent 在生产环境中执行错误的操作。没有审批流程，建模质量无人把关；没有版本锁定，审核中的模型被修改会导致审核失效。工作流是模型质量治理的最后一道防线，确保"发布出去的 Manifest 是经过至少两人确认的" |
| **AC-1** | GIVEN 李工完成语义/行为/事件建模 WHEN 点击"提交审核" THEN 模型状态从"草稿"变为"审核中"，所有建模元素进入只读模式，生成审核快照 |
| **AC-2** | GIVEN 模型处于"审核中"状态 WHEN 审核人(张工)查看模型 THEN 可在任意建模元素（聚合根/属性/行为/事件/校验规则）上添加批注，批注支持"通过/需修改/驳回"三种决议 |
| **AC-3** | GIVEN 审核人提交了"需修改"批注 WHEN 模型打回"草稿"状态 THEN 建模师可修改并重新提交审核，批注记录保留 |
| **AC-4** | GIVEN 所有审核批注均为"通过" WHEN 管理员(赵工)点击"批准发布" THEN 模型状态变为"已发布"，自动触发 Manifest 编译（US-A01），生成新版本号 |
| **AC-5** | GIVEN 模型已发布 WHEN 查看模型详情 THEN 可看到完整审批链：提交人、提交时间、审核人、审核批注、批准人、批准时间、生成的 Manifest 版本号 |
| **AC-6** | GIVEN 模型处于"审核中"或"已发布"状态 WHEN 尝试直接编辑 THEN 平台提示"模型已锁定，需创建新草稿版本才能编辑" |

### 8.1 Manifest 编译与版本管理

---

#### US-A01 [制品层] [P0-NOW] — 编译本体为 Agent Manifest

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 将完成建模的本体（语义+行为+事件+治理）一键编译为 Agent Manifest（版本化 JSON/YAML 快照）。Manifest 包含以下完整信息：①本体标识与版本（严格对应）②限界上下文定义 ③实体+属性+数据获取方式+聚合根标志 ④行为+参数+校验规则+声明式事件发布 ⑤领域事件+集成事件+Payload Schema ⑥角色+字段权限+Agent 沙箱配置 ⑦数据源元数据 |
| **Why** | Manifest 是本体平台的唯一交付物——它是一份机器可读的"合同"，告诉 Agent："这是业务世界的完整地图，这是你能走的路，这是你不能碰的禁区"。版本化意味着每个 Agent 绑定明确的本体版本，不会因模型更新而行为不一致 |
| **AC-1** | GIVEN 本体模型已完成建模 WHEN 点击"编译 Manifest" THEN 勾选要包含的层（语义/行为/事件/治理/数据源），选择输出格式(JSON/YAML) |
| **AC-2** | GIVEN 编译完成 WHEN 查看 Manifest 文件 THEN 文件包含完整的七段元数据（见 What 列表），结构符合预定义 Schema |
| **AC-3** | GIVEN Manifest 已生成 WHEN 修改本体（如新增一个行为） THEN 需重新编译生成新版本 Manifest，旧版本保留 |
| **AC-4** | GIVEN Manifest 编译失败 WHEN 查看错误信息 THEN 明确告知哪个元素缺少必填字段（如"行为'生产订单下达'缺少关联聚合根"） |
| **AC-5** | GIVEN Manifest 编译成功 WHEN 查看 Manifest 摘要 THEN 展示：版本号、包含的聚合根数/行为数/事件数/角色数、编译时间、校验通过状态 |

#### US-A02 [制品层] [P1-NEXT] — Manifest 版本对比

### 8.2 MCP 协议消费

---

#### US-A03 [制品层] [P0-NOW] — Agent 通过 MCP 查询本体

| 维度 | 内容 |
|------|------|
| **Who** | 王工(AID) — AI 应用开发工程师 |
| **What** | Agent 在运行时通过 MCP 工具 `query_ontology` 查询本体，获取当前意图相关的概念片段（如用户说"帮我查一下 PO-001 的状态"，Agent 查询本体获取"生产订单"的属性定义和可执行行为列表） |
| **Why** | 这就是三通道防御的软约束通道（通道二·运行时）——Agent 不是启动时加载整个 Manifest，而是运行时按需查询本体，将概念片段注入 LLM 上下文。这避免了上下文窗口爆炸，也保证了每次意图识别都能获取最新的本体定义 |
| **AC-1** | GIVEN Agent 收到用户意图 WHEN 调用 MCP `query_ontology(context="生产制造", concept="生产订单")` THEN 返回该概念的结构化定义：属性列表、允许的行为、状态机、校验规则摘要 |
| **AC-2** | GIVEN Agent 查询本体成功 WHEN LLM 生成响应 THEN 合规的响应中使用的术语和操作不超出本体返回的范围 |
| **AC-3** | GIVEN Agent 查询本体 WHEN Manifest 已更新到新版本 THEN 返回最新版本的定义（非缓存旧版本） |

#### US-A04 [制品层] [P1-NEXT] — MCP 工具白名单自动生成

| 维度 | 内容 |
|------|------|
| **Who** | 王工(AID) — AI 应用开发工程师 |
| **What** | 平台根据本体中定义的行为，自动生成 MCP 工具白名单配置。如"生产订单下达"行为 → 生成 MCP Tool `execute_production_order_release`，并自带参数校验 |
| **Why** | 手动配置 MCP 工具白名单易出错——漏配一个工具，Agent 调用失败；多配一个工具，安全风险 |
| **AC-1** | GIVEN 本体中有 6 个行为 WHEN 导出 MCP 配置 THEN 自动生成 6 个对应的 MCP Tool 定义，含参数 Schema |
| **AC-2** | GIVEN MCP 配置已生成 WHEN 校验层检查 Agent 调用 THEN 自动校验参数是否匹配行为定义的 Schema |

### 8.3 Agent 执行行为（三通道防御端到端）

---

#### US-A05 [制品层] [P0-NOW] — Agent 执行行为（三通道防御完整链路）

| 维度 | 内容 |
|------|------|
| **Who** | 王工(AID) — AI 应用开发工程师 |
| **What** | Agent 执行一个行为的完整链路——从用户自然语言到安全执行，经过三通道防御：①软约束（通道二）：用户说"下达工单 PO-001"→ LLM 意图识别 → 调 MCP `query_ontology` → 本体返回"生产订单"概念片段注入 LLM 上下文 → LLM 拆解为"执行行为：生产订单下达"②硬约束（通道一）：Agent 只能调用 MCP 工具白名单（如 `execute_production_order_release`），无 SQL/API/DB 直接访问权——结构上不可绕过 ③校验层（通道三）：Agent 输出经 Schema 白名单校验（实体/属性/枚举/行为前置条件/权限/聚合根入口），不合规即拒绝 |
| **Why** | 这是本体平台最核心的价值链——让 Agent 能安全地操作业务系统。三个通道缺一不可：没有软约束，Agent 不知道"生产订单"是什么；没有硬约束，Agent 可以绕过行为直接改数据库；没有校验层，Agent 可以下达不满足齐套校验的工单 |
| **AC-1** | GIVEN 用户输入自然语言"下达工单 PO-001" WHEN Agent 处理 THEN 第一步：调 `query_ontology` 获取概念片段 → 第二步：LLM 生成 MCP Tool 调用 `execute_production_order_release(order_id="PO-001")` → 第三步：MCP 网关校验白名单 → 第四步：校验层运行前置规则 → 第五步：执行行为并发布事件 |
| **AC-2** | GIVEN 物料齐套校验未通过 WHEN 执行到校验层 THEN 返回结构化违规信息，Agent 向用户展示"物料 MAT-001 缺口 50 件" |
| **AC-3** | GIVEN Agent 尝试绕过 MCP 直接调用 SQL WHEN 沙箱拦截 THEN 返回 403，Agent 无法执行任何数据库操作 |
| **约束** | 三通道的请求链路顺序固定：软约束(意图识别) → 硬约束(MCP白名单) → 校验层(执行前) → 执行。不可跳过、不可乱序 |

## 附录 B：AI Agent 沙箱补充说明

你在问题中问："AI Agent 沙箱（隔离执行环境）：我不理解，这个跟本体平台有什么关系？"

**沙箱不是本体平台之外的东西——它是本体定义的权限规则在运行时的物理执行容器。**

```
本体模型（定义层）                  AI Agent 沙箱（执行层）
┌─────────────────────┐           ┌─────────────────────────┐
│ "生产计划员"角色      │──编译──→  │ Agent 只能调用这 3 个工具   │
│ 只能执行: 创建/下达    │           │ 只能访问"生产订单"聚合根    │
│ 不能执行: 技术关闭    │           │ 看不到数据库连接串        │
│                     │           │ 看不到 API 密钥          │
│ 校验规则:            │──编译──→  │ 执行前自动运行校验        │
│ 物料齐套=100%        │           │ 不通过 → 拒绝执行         │
│                     │           │                         │
│ 字段级权限:          │──编译──→  │ Schema 返回时            │
│ "成本价"隐藏          │           │ 自动过滤"成本价"字段      │
│                     │           │                         │
│ 事件处理器矩阵:      │──编译──→  │ 仅响应授权场景下的处理器    │
│ MTS→自动补料         │           │ MTS场景才触发补料         │
│ MTO→按单领料         │           │ MTO场景才触发领料         │
└─────────────────────┘           └─────────────────────────┘
         ↑                                  ↑
    李工在平台里定义                 王工的 Agent 在这里面运行
```

本体是"宪法"，沙箱是"执法系统"。宪法写得好，执法才有依据；没有执法系统，宪法只是墙上的一张纸。

沙箱的每一项配置都**来自本体模型的编译结果**，不是独立于本体之外的另一套权限系统：

| 沙箱配置项 | 来自本体的哪一层 |
|-----------|---------------|
| 允许的 MCP 工具白名单 | 行为层 → 行为定义 |
| 允许访问的聚合根 | 语义层 → 聚合根定义 |
| 执行前校验规则 | 行为层 → 校验规则 |
| 字段过滤 | 治理层 → 字段级权限 |
| 操作频率限制 | 治理层 → 沙箱策略 |
| 事件处理器激活范围 | 事件层 → 处理器矩阵(场景×状态) |

### E.2 处理器详细定义

| # | 事件 | 场景 | 前置状态 | 处理行为 | 描述 |
|---|------|------|---------|---------|------|
| ① | 生产订单已下达 | MTS | 已下达 | 自动补料 | MTS 模式下工单下达后自动触发物料补库，无需人工领料 |
| ② | 生产订单已下达 | MTS | 已下达 | 自动领料 | MTS 模式下物料从库位自动调拨到线边仓 |
| ③ | 生产订单已下达 | MTO | 已下达 | 按单领料 | MTO 模式下需按销售订单逐笔领料 |
| ④ | 生产订单已下达 | MTO | 已下达 | 按单领料(二次) | MTO 模式补充领料 |
| ⑤ | 生产订单已部分投料 | MTS | 执行中 | 继续报工 | MTS 投料完成后允许开始报工 |
| ⑥ | 生产订单已部分投料 | MTO | 执行中 | 继续报工 | MTO 投料完成后允许开始报工 |
| ⑦ | 生产订单已部分报工 | MTS | 已报工 | 继续产出 | MTS 报工完成后允许成品产出 |
| ⑧ | 生产订单已部分报工 | MTO | 已报工 | 继续产出 | MTO 报工完成后允许成品产出 |
| ⑨ | 生产订单已部分产出 | MTS | 已产出 | 触发质检 | MTS 产出后触发质量检验（集成事件→质量管理） |

> **注意**：此矩阵为 MVP 阶段的简化版本。实际项目中，每个(事件,场景,状态)组合可能有不同的处理器，且可能存在"无处理器"的空位（表示该事件在该场景/状态下无需处理）。
