## 1. Ralph Loop 方法论

### 1.1 什么是 Ralph Loop

Ralph Loop 是 Jeff Patton 在 User Story Mapping 中引入的迭代产物发现循环。名字来源于《布偶秀》里只会问"为什么"的 Ralph——而"为什么"恰好是整个循环的灵魂。

```
        ┌─────────────────────┐
        │   Who（为目标用户）   │
        └─────────┬───────────┘
                  │
                  ▼
        ┌─────────────────────┐
        │  What（他们想达成什么）│←──────────┐
        └─────────┬───────────┘            │
                  │                        │
                  ▼                        │
        ┌─────────────────────┐            │
        │  Why（为什么这很重要）│────────────┘
        └─────────────────────┘    （每轮迭代回到 What，打磨粒度）
```

### 1.2 三圈打磨规则

每个 User Story 在本 PRD 中至少经过 **三轮 Ralph Loop**：

| 轮次 | 焦点 | 产出 |
|------|------|------|
| **R1（成立性）** | Who 是否真实？What 是否可验证？Why 是否触及业务指标？ | 故事一句话 |
| **R2（可验收）** | 加入 Acceptance Criteria（GIVEN/WHEN/THEN） | 验收标准 |
| **R3（约束与边界）** | 这个故事的完成不应依赖什么？必须兼容什么？ | 约束条件 |

### 1.3 故事格式约定

```
US-XXX [层] [优先级] — 故事标题
├─ Who： 具体角色（来自 Persona 列表）
├─ What：一句话描述 TA 想达成的结果（不是功能，是状态变化）
├─ Why：  底层动机（可追溯到业务指标或痛点指标）
├─ AC：   GIVEN/WHEN/THEN 验收标准
└─ 约束： 依赖项 / 非依赖项 / 兼容性要求
```

### 2.2 详细画像

#### Persona A：张工 — 业务领域专家（SME）

- **年龄/背景**：45 岁，20 年制造业经验，历任车间主任→生产部长→精益顾问
- **技术水平**：会用 ERP（SAP），不会写代码，不会 SQL。能画业务流程图
- **日常工作**：制定生产策略、审核工单、协调物料、处理异常
- **与平台的关系**：他是**知识的提供者**——他定义"什么是物料齐套""什么时候工单能下达"。他的知识是本体模型的原材料
- **关键指标关注**：准时完工率、工单平均完成时间、物料周转天数
- **最大痛点**：他的业务知识全部在脑子里和 Excel 里。他退休/调岗的那一天，这些规则就丢了

#### Persona B：李工 — 本体建模工程师（Ontology Engineer）

- **年龄/背景**：32 岁，计算机专业，5 年 DDD 实践，做过微服务架构
- **技术水平**：熟练 DDD 战术模式（聚合/实体/值对象/领域事件），会写 SQL/API 集成代码
- **日常工作**：需求分析→领域建模→设计聚合→定义接口→Code Review
- **与平台的关系**：他是平台的**核心用户**——他把张工的业务知识转化为结构化的本体模型。建模质量直接决定 Agent 能多安全地操作业务系统
- **最大痛点**：每次新项目从零开始画 ER 图、写接口文档。之前项目定义的"生产订单"模型无法复用，因为散落在代码、Wiki、飞书文档里

#### Persona C：王工 — AI 应用开发工程师（AI Developer）

- **年龄/背景**：28 岁，全栈+AI，2 年 LLM 应用开发经验
- **技术水平**：Python/Node.js、LangChain/LlamaIndex、MCP 协议、Prompt Engineering
- **日常工作**：开发企业内部 AI 助手（如"生产计划助手""物料查询助手"）
- **与平台的关系**：他是本体的**消费者**——他通过 Manifest 和 MCP 协议让 AI Agent 安全地操作业务系统
- **最大痛点**：Agent 幻觉导致它调了不该调的 API（比如把未审核的工单直接下达了）。他不知道 Agent "应该能做什么、不能做什么"——因为没有一份机器可读的"权限清单"

#### Persona D：赵工 — 平台管理员（Platform Admin）

- **年龄/背景**：35 岁，运维+安全背景，负责企业 IT 基础设施
- **技术水平**：Linux/容器化、IAM、安全审计、合规
- **日常工作**：用户管理、权限分配、系统监控、安全审计
- **与平台的关系**：他是平台的**守门人**——他管理谁能访问什么、Agent 能在哪个沙箱运行、数据源连接是否安全
- **最大痛点**：权限配置散落在多个系统（ERP 的、数据库的、API 网关的），出事时审计链路断裂——不知道是"人改的"还是"Agent 改的"

#### Persona E：陈工 — 运营分析师（Operations Analyst）

- **年龄/背景**：30 岁，工业工程专业，擅长数据分析
- **技术水平**：熟练 BI 工具（Power BI/Tableau）、会写基础 SQL
- **日常工作**：制作生产报表、分析 KPI 趋势、识别瓶颈
- **与平台的关系**：他是**指标和事件的消费者**——他用本体定义的指标口径和事件数据来确保"所有人看的是同一个数字"
- **最大痛点**：ERP 系统报的"准时完工率"是 78%，MES 报的是 82%，BI 报表又算出来 75%。三个系统三个数，他不知道信哪个——因为没人说清楚"准时"到底怎么算

## 3. 用户旅程主干（Backbone）

Backbone 是用户从"我有一个业务场景"到"AI Agent 安全运行在生产环境"的完整旅程。

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  ① 领域定义   │───→│  ② 语义建模   │───→│  ③ 行为建模   │───→│  ④ 事件建模   │───→│  ⑤ 治理配置   │
│  (张工主导)   │    │  (李工主导)   │    │  (李工主导)   │    │  (李工主导)   │    │  (赵工主导)   │
│  划定边界      │    │  结构化知识    │    │  定义规则     │    │  编排联动     │    │  权限与安全   │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
                                                                                      │
                                                                                      ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ ⑩ 运营分析   │←───│  ⑨ 事件消费   │←───│  ⑧ Agent运行  │←───│  ⑦ MCP消费    │←───│  ⑥ 制品发布   │
│  (陈工主导)   │    │  (陈工主导)   │    │  (王工主导)   │    │  (王工主导)   │    │  (李工主导)   │
│  指标与洞察   │    │  联动与追溯   │    │  安全执行     │    │  集成开发     │    │  版本化输出   │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

### 3.1 十步简介

| 步骤 | 主干名 | 主导角色 | 一句话 |
|------|--------|---------|--------|
| ① | 领域定义 | 张工(SME) | 划定限界上下文和业务场景的边界 |
| ② | 语义建模 | 李工(OE) | 将业务概念转化为聚合根/对象/属性/关系 |
| ③ | 行为建模 | 李工(OE) | 定义行为、校验规则、指标公式、事务边界 |
| ④ | 事件建模 | 李工(OE) | 定义领域事件、集成事件、事件路由和处理器 |
| ⑤ | 治理配置 | 赵工(Admin) | 配置角色权限、字段级权限、AI Agent 沙箱策略 |
| ⑥ | 制品发布 | 李工(OE) | 编译本体为版本化 Agent Manifest |
| ⑦ | MCP 消费 | 王工(AID) | Agent 通过 MCP 协议查询本体、获取工具白名单 |
| ⑧ | Agent 运行 | 王工(AID) | Agent 在执行时受三通道防御保护 |
| ⑨ | 事件消费 | 陈工(OA) | 领域事件触发跨系统联动、更新指标 |
| ⑩ | 运营分析 | 陈工(OA) | 基于统一指标口径分析生产运营表现 |

## 4. User Stories — 语义层

> 语义层覆盖主干步骤 ①②：领域定义 + 语义建模

### 4.1 限界上下文管理

---

#### US-S01 [语义层] [P0-NOW] — 创建限界上下文并绑定本体

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 在平台上创建一个名为"生产制造"的限界上下文，它与一个本体 1:1 绑定。创建完成后，这个上下文成为后续所有建模操作的命名空间 |
| **Why** | 限界上下文是 DDD 战略设计的核心——不先划定边界，后续"物料""生产订单"这些概念属于哪个领域都不清楚。1:1 绑定本体意味着：**一个上下文 = 一个独立可发布的 Agent Manifest** |
| **AC-1** | GIVEN 用户进入平台首页 WHEN 点击"新建限界上下文" THEN 弹出表单，要求填写：上下文名称、描述、所属领域标签 |
| **AC-2** | GIVEN 已填写上下文名称"生产制造" WHEN 点击"创建" THEN 平台自动创建一个同名本体并绑定，显示在上下文列表中 |
| **AC-3** | GIVEN 已存在名为"生产制造"的上下文 WHEN 尝试创建同名上下文 THEN 平台提示"该限界上下文已存在"，不允许重复 |
| **AC-4** | GIVEN 上下文已创建 WHEN 查看上下文详情 THEN 可看到：绑定的本体 ID、创建时间、包含的聚合根数量（初始为 0） |
| **约束** | 限界上下文名称在平台内全局唯一；删除上下文需要先清空其下的所有模型元素 |

#### US-S02 [语义层] [P1-NEXT] — 在上下文中定义业务场景

| 维度 | 内容 |
|------|------|
| **Who** | 张工(SME) — 业务领域专家 |
| **What** | 在"生产制造"限界上下文下，定义两个业务场景："面向库存生产"和"面向订单生产"。每个场景可以有不同的行为规则和校验逻辑 |
| **Why** | 同一个"生产订单"在不同业务场景下行为不同——MTS（面向库存生产）允许自动补货触发，MTO（面向订单生产）必须严格按销售订单驱动。场景是行为规则的分类维度 |
| **AC-1** | GIVEN 进入"生产制造"上下文 WHEN 点击"新建业务场景" THEN 创建"面向库存生产"和"面向订单生产"两个场景 |
| **AC-2** | GIVEN 场景已创建 WHEN 在语义建模中选择聚合根 THEN 可指定该聚合根适用于哪些场景（支持多选） |
| **AC-3** | GIVEN 后续定义行为规则 WHEN 关联规则到场景 THEN 规则仅在指定场景下生效 |

### 4.2 聚合根与对象建模

---

#### US-S03 [语义层] [P0-NOW] — 定义聚合根

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 在"生产制造"上下文中定义四个聚合根：生产订单、物料、BOM、工艺路线。每个聚合根是其他对象（实体、值对象）的入口点 |
| **Why** | 聚合根是 DDD 事务一致性的边界，也是 AI Agent 访问业务数据的唯一入口——**Agent 不能绕过聚合根直接操作子对象**。这是 AI 安全的第一道防线 |
| **AC-1** | GIVEN 进入上下文建模界面 WHEN 点击"添加聚合根" THEN 弹出表单：名称(必填)、标识符(中文名+英文code)、描述、所属场景 |
| **AC-2** | GIVEN 已定义"生产订单"聚合根 WHEN 添加子对象（如"工序"） THEN 子对象自动挂载在该聚合根下，不允许独立存在 |
| **AC-3** | GIVEN 定义了多个聚合根 WHEN 后续定义关系类型 THEN 允许跨聚合根建立关系（如"生产订单→物料"） |
| **AC-4** | GIVEN 聚合根已创建 WHEN 查看数据血缘图 THEN 可看到该聚合根依赖的数据源和获取方式 |
| **约束** | 聚合根数量下限：一个限界上下文至少 1 个；上限：无硬限制，但建议单个上下文 ≤ 10 个聚合根以保证可维护性 |

#### US-S04 [语义层] [P0-NOW] — 定义对象类型与属性

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 为"生产订单"聚合根定义对象类型（如"生产订单头""工序""物料组件"），每个对象类型包含属性定义（字段名、类型、必填/可选、枚举值） |
| **Why** | 对象属性是 Agent Manifest 中 Schema 白名单的基础——校验层会用它来拦截 Agent 的非法输出。属性定义的质量直接决定 Agent 安全 |
| **AC-1** | GIVEN 选中"生产订单"聚合根 WHEN 点击"添加子对象" THEN 可定义：对象名称、对象类型(实体/值对象/接口抽象)、描述 |
| **AC-2** | GIVEN 子对象已创建 WHEN 添加属性 THEN 每条属性包含：字段名(中英文)、数据类型(STRING/INT/DECIMAL/DATE/ENUM/BOOLEAN/JSON)、是否必填、默认值、枚举值列表(当类型为ENUM时) |
| **AC-3** | GIVEN 属性已定义 WHEN 后续定义字段级权限 THEN 可将敏感属性（如"成本价"）标记为需授权字段 |
| **AC-4** | GIVEN 属性列表中包含多行 WHEN 拖拽调整顺序 THEN 属性顺序持久化保存 |

#### US-S05 [语义层] [P1-NEXT] — 定义值对象

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 定义可复用的值对象："地址""计量单位""金额"。值对象没有独立 ID，以整体形式嵌入到实体属性中 |
| **Why** | 值对象避免"基本类型偏执"——"金额"不应该只是一个 DECIMAL，它应该包含币种和精度。"地址"不是三个 VARCHAR，它是省市区+详细地址的组合语义 |
| **AC-1** | GIVEN 在对象建模界面 WHEN 选择"新建值对象" THEN 定义值对象名称及内部属性组合 |
| **AC-2** | GIVEN 值对象"金额"已定义（含 amount + currency） WHEN 实体属性选择类型为"金额" THEN 该属性自动继承金额的复合结构 |
| **AC-3** | GIVEN 值对象被修改 WHEN 引用该值对象的实体查看属性 THEN 属性结构自动同步更新 |
| **约束** | 值对象创建为全局级别（跨上下文复用），修改需提示影响范围 |

#### US-S06 [语义层] [P1-NEXT] — 定义对象状态与状态机

| 维度 | 内容 |
|------|------|
| **Who** | 张工(SME) + 李工(OE) — 业务专家 + 建模师 |
| **What** | 为"生产订单"定义其生命周期状态：创建→已下达→执行中→已报工→已产出→技术关闭。每个状态转换有前置条件和触发行为 |
| **Why** | 状态机是业务规则的结构化表达——"工单在'创建'状态下不能报工"这个规则不是写在 if-else 里，而是定义在模型中，Agent 执行前校验层就能拦截 |
| **AC-1** | GIVEN 选中"生产订单"对象 WHEN 打开"状态定义"面板 THEN 可添加状态节点：状态名、状态码、是否为终态、是否为初态 |
| **AC-2** | GIVEN 已定义 6 个状态 WHEN 定义状态转换 THEN 选择"从状态→到状态"，关联触发行为（如"创建→已下达"需要"下达工单"行为） |
| **AC-3** | GIVEN 状态机已配置 WHEN 校验层检查 Agent 输出 THEN 非法状态跳转（如"已关闭→执行中"）被拒绝 |

### 5.2 校验规则

---

#### US-B03 [行为层] [P0-NOW] — 定义校验规则并关联行为

| 维度 | 内容 |
|------|------|
| **Who** | 张工(SME) 提供规则知识，李工(OE) 负责结构化录入 |
| **What** | 定义规则 #1（物料齐套校验）：BOM 中所有物料的库存余量 ≥ 工单需求量，否则拦截下达。定义规则 #2（工艺路线校验）：成品物料必须有且仅有一条有效工艺路线，且工序顺序不能有环 |
| **Why** | 校验规则是 AI 安全的第二道防线（校验层）——Agent 执行"下达工单"前，规则引擎自动运行这些规则。规则定义在模型中而非硬编码在代码里，意味着非技术人员（张工）也可以参与规则维护 |
| **AC-1** | GIVEN 进入规则管理界面 WHEN 点击"新建规则" THEN 定义：规则名称、规则类型(前置校验)、规则描述(自然语言)、规则表达式(结构化)、违规提示文案 |

| **AC-3** | GIVEN 规则已定义 WHEN 关联到行为"生产订单下达" THEN 规则作为前置条件挂载，执行顺序可调整 |
| **AC-4** | GIVEN 规则校验失败 WHEN Agent 尝试执行 THEN 返回结构化的违规信息：{rule: "物料齐套校验", failed_items: [{物料编码: "MAT-001", 缺口: 50}], message: "物料 MAT-001 库存不足，缺口 50 件"} |
| **约束** | 校验规则必须返回结构化结果（JSON），不能只返回"校验失败"文本——Agent 需要足够信息来向用户解释"为什么失败"和"缺什么" |

#### US-B04 [行为层] [P2-LATER] — 规则版本管理与灰度发布

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) + 张工(SME) |

### 7.2 AI Agent 沙箱

---

#### US-G04 [治理层] [P0-NOW] — 配置 AI Agent 沙箱

| 维度 | 内容 |
|------|------|
| **Who** | 赵工(Admin) — 平台管理员 |
| **What** | 为 AI Agent 配置沙箱执行环境：Agent 默认无任何权限，必须在沙箱配置中显式授权它调用哪些 MCP 工具、访问哪些聚合根、执行哪些行为。沙箱还有一个"最大操作频率"限制（防止 Agent 意外循环调用） |
| **Why** | 这就是你之前问的"沙箱跟本体平台有什么关系"——沙箱的权限白名单**来自本体模型**。本体定义了"Agent 能做什么"，沙箱是执行这个定义的容器。Agent 在沙箱里看不到数据库、看不到 API 密钥、没有文件系统——它只能调用你显式授权的那几个 MCP 工具 |
| **AC-1** | GIVEN 进入沙箱配置 WHEN 创建新沙箱 THEN 选择：绑定的 Manifest 版本、Agent 角色、允许的 MCP 工具列表、允许的聚合根、允许的行为列表 |
| **AC-2** | GIVEN 沙箱已创建 WHEN Agent 尝试调用未授权的工具 THEN 沙箱返回 403 错误，并在审计日志中记录 |
| **AC-3** | GIVEN 沙箱已配置频率限制(如 10次/秒) WHEN Agent 超过频率 THEN 沙箱返回 429 限流错误 |
| **AC-4** | GIVEN 沙箱中的 Agent 执行操作 WHEN 查看审计日志 THEN 可追溯到：哪个 Agent、哪个沙箱、哪个行为、什么时间、操作了什么数据 |
| **约束** | 沙箱是最小权限原则的物理实现——不是"默认允许，黑名单拦截"，而是"默认拒绝，白名单放行" |

## 8. User Stories — 制品管理层

> 制品层覆盖主干步骤 ⑥⑦：制品发布 + MCP 消费

### 8.1 Manifest 编译与版本管理

---

#### US-A01 [制品层] [P0-NOW] — 编译本体为 Agent Manifest

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) — 本体建模工程师 |
| **What** | 将完成建模的本体（语义+行为+事件+治理）一键编译为 Agent Manifest（版本化 JSON/YAML 快照）。Manifest 包含：本体标识+版本、限界上下文定义、实体+属性+数据获取方式+聚合根标志、行为+参数+校验规则+声明式事件发布、领域事件+集成事件+Payload Schema、角色+字段权限+Agent 沙箱配置、数据源元数据 |
| **Why** | Manifest 是本体平台的唯一交付物——它是一份机器可读的"合同"，告诉 Agent："这是业务世界的完整地图，这是你能走的路，这是你不能碰的禁区"。版本化意味着每个 Agent 绑定明确的本体版本，不会因模型更新而行为不一致 |
| **AC-1** | GIVEN 本体模型已完成建模 WHEN 点击"编译 Manifest" THEN 勾选要包含的层（语义/行为/事件/治理/数据源），选择输出格式(JSON/YAML) |
| **AC-2** | GIVEN 编译完成 WHEN 查看 Manifest 文件 THEN 文件包含完整的五层元数据，结构符合预定义 Schema |
| **AC-3** | GIVEN Manifest 已生成 WHEN 修改本体（如新增一个行为） THEN 需重新编译生成新版本 Manifest，旧版本保留 |
| **AC-4** | GIVEN Manifest 编译失败 WHEN 查看错误信息 THEN 明确告知哪个元素缺少必填字段（如"行为'生产订单下达'缺少关联聚合根"） |

#### US-A02 [制品层] [P1-NEXT] — Manifest 版本对比

| 维度 | 内容 |
|------|------|
| **Who** | 李工(OE) + 王工(AID) |
| **What** | 对比两个 Manifest 版本之间的差异：新增了哪些行为、修改了哪些校验规则、删除了哪些属性 |
| **Why** | 王工在升级 Agent 依赖的 Manifest 版本时，需要知道"这个版本变了什么"——如果新增了必填参数，他的 Agent 代码需要适配 |
| **AC-1** | GIVEN Manifest v1.0 和 v1.1 都存在 WHEN 点击"版本对比" THEN 展示 diff：新增/修改/删除的元素，按层分组 |

#### US-A03 [制品层] [P0-NOW] — Agent 通过 MCP 查询本体

| 维度 | 内容 |
|------|------|
| **Who** | 王工(AID) — AI 应用开发工程师 |
| **What** | Agent 在运行时通过 MCP 工具 `query_ontology` 查询本体，获取当前意图相关的概念片段（如用户说"帮我查一下 PO-001 的状态"，Agent 查询本体获取"生产订单"的属性定义和可执行行为列表） |
| **Why** | 这就是三通道防御的软约束通道——Agent 不是启动时加载整个 Manifest，而是运行时按需查询本体，将概念片段注入 LLM 上下文。这避免了上下文窗口爆炸，也保证了每次意图识别都能获取最新的本体定义 |
| **AC-1** | GIVEN Agent 收到用户意图 WHEN 调用 MCP `query_ontology(context="生产制造", concept="生产订单")` THEN 返回该概念的结构化定义：属性列表、允许的行为、状态机、校验规则摘要 |
| **AC-2** | GIVEN Agent 查询本体成功 WHEN LLM 生成响应 THEN 合规的响应中使用的术语和操作不超出本体返回的范围 |
| **AC-3** | GIVEN Agent 查询本体 WHEN Manifest 已更新到新版本 THEN 返回最新版本的定义（非缓存旧版本） |

#### US-A04 [制品层] [P1-NEXT] — MCP 工具白名单自动生成

| 维度 | 内容 |
|------|------|
| **Who** | 王工(AID) — AI 应用开发工程师 |
| **What** | 平台根据本体中定义的行为，自动生成 MCP 工具白名单配置。如"生产订单下达"行为 → 生成 MCP Tool `execute_production_order_release`，并自带参数校验 |

## 附录 B：关于 AI Agent 沙箱的补充说明

你在问题中问："AI Agent 沙箱（隔离执行环境）：我不理解，这个跟本体平台有什么关系？"

答案在上文 US-G04 中已详细展开，这里做一个浓缩总结：

```
本体模型（定义层）                  AI Agent 沙箱（执行层）
┌─────────────────────┐           ┌─────────────────────────┐
│ "生产计划员"角色      │──编译──→  │ Agent 只能调用这 3 个工具   │
│ 只能执行: 创建/下达    │           │ 只能访问"生产订单"聚合根    │
│ 不能执行: 技术关闭    │           │ 看不到数据库连接串        │
│                     │           │ 看不到 API 密钥          │
│ 校验规则:            │──编译──→  │ 执行前自动运行校验        │
│ 物料齐套=100%        │           │ 不通过 → 拒绝执行         │
│                     │           │                         │
│ 字段级权限:          │──编译──→  │ Schema 返回时            │
│ "成本价"隐藏          │           │ 自动过滤"成本价"字段      │
└─────────────────────┘           └─────────────────────────┘
         ↑                                  ↑
    李工在平台里定义                 王工的 Agent 在这里面运行
```

**沙箱不是本体平台之外的东西——它是本体定义的权限规则在运行时的物理执行容器。** 本体是"宪法"，沙箱是"执法系统"。宪法写得好，执法才有依据；没有执法系统，宪法只是墙上的一张纸。
