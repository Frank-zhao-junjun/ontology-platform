## 1. 产品概述

### 1.1 产品定位

**本体建模即服务（Ontology Modeling as a Service）**——企业 AI 平台的"业务语义中台"。

本体建模平台是连接企业业务系统与大模型智能能力的核心中间层。它将分散在 ERP、CRM、MES 等系统中的业务数据，转化为 AI 可理解、可执行、可审计的结构化语义模型（本体），解决大模型在企业级应用中"不懂业务、幻觉频发、执行越权"三大困境。

#### 1.1.1 Non-Goals —— 本平台明确不做什么

| 不做 | 原因 | 替代方案 |
|------|------|---------|
| **不存储实例数据** | 平台是语义中台，不是数据中台 | 消费方（AI Agent / 应用）自行调用数据源 |
| **不做知识图谱推理** | 本体是 Schema 层，推理是下游能力 | 知识图谱平台读取本体 Schema 后自行推理 |
| **不做 ETL / 数据同步** | 平台不执行数据拉取或定时同步 | 数据中台 / CDC 工具负责 |
| **不做 BI 报表** | 指标定义归本体，可视化归 BI 工具 | BI 工具读取指标定义自行渲染 |
| **不做 LLM 训练 / Fine-tuning** | 平台提供语义上下文（RAG），不训练模型 | AI 平台（Dify, LangChain, Coze）负责模型层 |
| **不做工作流引擎** | 行为编排在 Agent 层，平台只管概念定义 | 工作流引擎（Temporal, Airflow）编排调用 |
| **不做 API Gateway** | 平台暴露本体 API，路由/限流/认证由网关管理 | 企业 API Gateway（Kong, APISIX） |
| **不做低代码/无代码应用搭建** | 本体是语义模型，不是应用开发平台 | 低代码平台（明道云, 简道云, Retool） |

#### 1.1.2 双产品边界——设计台与平台的关系

本 PRD 覆盖**两个协作产品**，共享五层本体内容模型，但职责与运行时边界不同。详细双产品边界说明见 [User Story Map §0](./PRD-本体建模平台-UserStoryMap-v1.2.md#0-双产品边界与-manifest-交接)。

| | **项目1：Ontology 设计台** | **项目2：ontology-platform（本文档主体）** |
|--|---------------------------|------------------------------------------|
| 代码 | `Ontology` 仓库（Next.js） | `ontology-platform` 仓库 |
| 用户 | 李工(OE)、张工(SME) — 建模与评审 | 李工、赵工、王工 — 发布、治理、Agent 消费 |
| 交付 | 本地 persist 草稿、预览、**导出 YAML** | 导入 YAML → 校验 → **发布** → MCP |
| 不做 | 权威存储、发布、MCP、运行时沙箱 | AI 辅助草稿生成（可由设计台承担） |

**唯一交接契约**：`OntologyManifest`（`apiVersion: ontology.platform/v1`），规范见 [ontology-manifest-spec.md](./ontology-manifest-spec.md)。

```
设计台(导出 YAML) ──YAML──► 平台(导入校验) ──published──► MCP(Agent 消费)
```

> **本文档主体为项目2（平台）**，覆盖运行期能力（MCP、三通道防御、Agent 沙箱）。设计台的建模期 UI 能力详见 User Story Map 附录 F（US-D01～D05）。

### 1.4 四层架构概览

平台核心采用四层架构模型，每层对应一类业务语义：

| 层级 | 定位 | 核心内容 | USM 覆盖 |
|------|------|---------|----------|
| **语义层** | 业务「名词」— 定义业务是什么 | 限界上下文、聚合根、对象属性、关系类型、值对象、状态机、数据源、数据血缘 | S01–S10 |
| **行为层** | 业务「动词」— 定义能做什么 | 行为定义、校验规则、事务边界、副作用、指标定义、参数模板、规则灰度 | B01–B07 |
| **事件层** | 业务「信号」— 定义发生了什么 | 领域事件、集成事件、事件路由、事件处理器矩阵、事件存储 | E01–E05 |
| **治理层** | 业务「权限」— 定义谁能做什么 | 角色权限、字段级权限、条件权限、AI Agent 沙箱、建模工作流 | G01–G05 |

**制品层**（第五层）是四层的编译产物——将本体模型编译为版本化的 Agent Manifest，供 AI Agent 通过 MCP 协议消费。

### 1.5 背景与驱动力

#### 1.5.1 企业 AI 落地的"最后一公里"困境

当前企业 AI 应用面临三重鸿沟：

```
┌─────────────────────────────────────────────────────────────────┐
│  语义鸿沟          幻觉鸿沟           执行鸿沟                    │
│  ─────────        ─────────         ─────────                    │
│  AI 不知道         AI 可能生成        AI 只能建议                 │
│  "客户"在 CRM      不存在的订单       不能操作                    │
│  中对应哪张表      编号               ERP 系统                    │
│                                                                    │
│  AI 不理解          AI 可能承诺        AI 操作后无                 │
│  "状态=3"代表      不存在的库存       法追溯审计                   │
│  什么业务含义                                                        │
└─────────────────────────────────────────────────────────────────┘
```

**本体建模平台的答案**：在 LLM 和业务系统之间插入一个结构化的语义层——本体告诉 AI "业务世界有哪些概念、它们之间什么关系、你能做什么操作、每个操作有什么规则"。

#### 1.5.2 为什么是"本体"而不是"Prompt"或"Fine-tuning"

| 方案 | 问题 |
|------|------|
| **Prompt Engineering** | 上下文窗口有限，复杂业务规则写不下；每次对话都要重写，不一致 |
| **Fine-tuning** | 业务规则变更需重新训练；无法保证 100% 遵守约束；黑盒不可审计 |
| **RAG + 文档** | 非结构化文档无法精确约束 Agent 行为；Agent 仍可能"理解偏差" |
| **本体模型（本方案）** | 结构化、机器可读、版本化、可审计；编译为 Manifest 后 Agent 行为受严格约束 |

## 2. 市场分析

### 2.1 痛点量化

| 痛点 | 表现 | 量化影响 |
|-----|------|---------|
| **语义鸿沟** | AI 不理解业务术语，答非所问 | 客服满意度 <60%；业务查询需人工中转 |
| **幻觉失控** | AI 生成不存在的订单/库存/承诺 | 业务错误率 >5%；客诉风险 |
| **执行闭环缺失** | AI 只能给建议，不能安全执行 | 人工介入率 >30%；效率未实质提升 |
| **知识流失** | 业务专家离职带走隐性知识 | 新人上手周期 >3 个月；规则一致性差 |

### 2.2 竞品分析

#### 2.2.1 核心竞品对比

| 维度 | Palantir Foundry | 自研方案 | **本体建模平台** |
|-----|-----------------|---------|----------------|
| **本体建模** | ★★★★★ 成熟 | ★★☆☆☆ 需从头构建 | ★★★★☆ 开箱即用 |
| **AI Agent 安全** | ★★★★☆ 内置沙箱 | ★★☆☆☆ 需自建 | ★★★★★ 三通道防御 |
| **部署灵活性** | ★★☆☆☆ 仅 SaaS | ★★★★★ 完全可控 | ★★★★☆ 支持私有化 |
| **成本（年）** | 数千万元 | 数百万元（人力） | 数十万–百万元 |
| **实施周期** | 6–12 个月 | 6–18 个月 | 1–3 个月 |
| **国产化适配** | ★☆☆☆☆ 不支持 | ★★★★★ 可定制 | ★★★★★ 原生支持 |
| **MCP 协议** | 不支持 | 可开发 | 原生支持 |
| **事件驱动架构** | ★★★★★ 成熟 | ★★★☆☆ 需搭建 | ★★★★☆ 内置 |

#### 2.2.2 Palantir Foundry 深度分析

**优势**：四层架构（Ontology → Actions → Events → Security）设计成熟，全球头部客户验证。

**劣势**：
- 价格极高（年费千万级），仅适合超大型企业
- 纯 SaaS，不支持私有化部署（数据出境合规风险）
- 对国产大模型和国内企业系统（用友/金蝶/华为 ERP）集成支持弱
- 实施周期长，本地化服务能力有限

#### 2.2.3 我们的机会窗口

Palantir 在中国市场的空白 + 国产大模型生态的爆发 + 企业对 AI 安全执行的刚需 = **本体建模平台的蓝海窗口**。

### 2.3 差异化定位

| 差异化点 | 描述 |
|---------|------|
| **三通道防御** | 软约束(MCP 查询本体) → 硬约束(MCP 工具白名单) → 校验层(Schema+规则+权限)——业界首创的 Agent 安全执行架构 |
| **私有化部署** | 支持完全私有化，满足金融/政务/军工数据合规 |
| **国产化适配** | 原生支持国产大模型（文心/通义/智谱）、国产数据库、国产 OS |
| **MCP 原生** | 编译本体为 Manifest → MCP Server 自动生成工具白名单，与 Dify/Coze/LangChain 无缝对接 |
| **快速落地** | 预置制造域参考模型（附录 C），1 周 POC，1–3 月上线 |
| **性价比** | 相比 Palantir 降低 80%+ 成本 |

### 3.2 语义层 — 定义业务「名词」+ 数据在哪（S01–S10）

**核心能力**：限界上下文 1:1 绑定本体 → 聚合根作为数据访问唯一入口 → 对象属性定义 + 状态机 + 关系类型（组合/聚合/引用/依赖）→ 数据源仅记录获取元数据（SQL/API/MCP），平台不存储实例数据。

**MVP 关键 Story**：
| ID | 能力 | 说明 |
|----|------|------|
| S01 | 创建限界上下文 | 一个上下文 = 一个独立可发布的 Manifest |
| S03 | 定义聚合根 | Agent 访问数据的唯一入口——不能绕过聚合根直接操作子对象 |
| S04 | 对象类型与属性 | 属性定义是 Schema 白名单的基础——校验层用它拦截 Agent 非法输出 |
| S07 | 关系类型 | 组合/聚合/引用/依赖四种关系 + 基数(1:1/1:N/N:M) |
| S08 | 数据源与获取方式 | SQL/API/MCP 三种方式，仅记录元数据不执行调用 |

### 3.3 行为层 — 定义业务「动词」+ 规则（B01–B07）

**核心能力**：行为是 Agent 的执行元语 → 行为自带校验规则和事件发布声明 → 事务边界确保原子性 → 副作用异步处理（与核心事务解耦）→ 指标定义统一口径。

**MVP 关键 Story**：
| ID | 能力 | 说明 |
|----|------|------|
| B01 | 行为定义 | 支持**主动调用**和**事件驱动**两种模式，同一行为校验逻辑一致 |
| B03 | 校验规则 | 返回结构化违规信息（`failed_items` + `message`），Agent 可向用户解释原因 |
| B05 | 指标定义 | 平台只存储指标元数据（公式+数据源），不执行计算——口径统一，消费方自行计算 |
| B06 | 事务边界 | 行为→状态转换→事件发布为原子操作，失败全部回滚 |

### 3.4 事件层 — 定义业务「信号」+ 联动（E01–E05）

**核心能力**：领域事件由行为声明式发布 → 事件路由定义分发规则 → 事件处理器矩阵（事件×场景×状态→处理行为）→ 集成事件支持跨上下文传播 → 事件存储支持溯源。

**关键设计**：
- **E03/E04 是建模期定义**：不依赖事件存储即可完成路由和处理器的配置
- **MVP 使用进程内事件通道**：E02 事件存储为运行期持久化，MVP 可先使用同步内存通道
- **因果链**：行为执行 → 状态转换 → 领域事件发布 → 事件处理器执行（S06→E01→E04）

### 3.5 治理层 — 定义「谁能做什么」+ 流程（G01–G05）

**核心能力**：RBAC 角色权限（上下文级覆盖全局默认）→ 字段级权限（敏感字段对无权限角色不可见）→ AI Agent 沙箱（最小权限 + 白名单 + 频率限制）→ 建模工作流（草稿→审核→发布，强制双人复核）。

**MVP 关键 Story**：
| ID | 能力 | 说明 |
|----|------|------|
| G01 | 角色与对象级权限 | 权限挂在限界上下文上，上下文级可覆盖全局默认 |
| G04 | AI Agent 沙箱 | 默认零权限，白名单放行；配置来自本体模型编译结果 |
| G05 | 建模工作流 | 草稿→审核→发布，强制双人复核，确保 Manifest 质量 |

### 3.6 制品管理层 — 编译本体为 Agent Manifest（A01–A05）

**核心能力**：本体一键编译为版本化 Manifest（七段元数据）→ 版本对比识别破坏性变更 → MCP **`resolve_intent` + `query_ontology`** 运行时解析意图并注入本体定义 → MVP 统一 **`execute_action`** 执行（P1 可选每行为一 Tool，US-A04）→ **三通道防御端到端生效**。

**三通道防御链路**（US-A05，平台最核心的价值链）：

```
用户："下达工单 PO-001"
    │
    ▼
Step 1 — resolve_intent（语义路由：模糊→精确）
    │  输入："下达工单 PO-001"
    │  输出：{context:"manufacturing", aggregate:"ProductionOrder",
    │         behavior:"release_production_order"}
    │
    │  ← LLM sanity check: "ProductionOrder 没错，release 是下达"
    ▼
Step 2 — query_ontology（通道二·软约束：精确概念→完整定义）
    │  输入：context="manufacturing", concept="ProductionOrder"
    │  输出：属性列表 + 状态机 + 校验规则 + 允许行为 → 注入 LLM 上下文
    │
    ▼
Step 3 — execute_action（通道一·硬约束 + 通道三·校验层）
    │  LLM 生成 Tool 调用 → MCP 白名单校验（通道一）
    │  → Schema 白名单校验 → 前置规则校验（物料齐套？）
    │  → 权限校验 → 聚合根入口校验（通道三）
    │
    ▼
执行行为 + 发布事件 + 记录审计日志
```

### 4.2 核心技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **建模语言** | 自研 DSL + JSON Schema | 兼顾结构化校验与可读性；OWL/RDF 太重 |
| **Manifest 格式** | JSON + JSON Schema 校验 | Agent 可直接解析；版本化管理 |
| **事件通道** | MVP: 进程内同步 / 生产: Kafka | 渐进式架构——先验证逻辑，再升级持久化 |
| **权限模型** | RBAC + ABAC（条件权限） | RBAC 覆盖 90% 场景，ABAC 覆盖时间/组织维度 |
| **MCP 协议** | 原生 MCP Server | 与 Dify/Coze/LangChain 等主流 AI 平台无缝对接 |
| **存储** | PostgreSQL（元数据）+ 对象存储（Manifest） | 成熟可靠，支持 JSONB 灵活查询 |

### 4.3 安全架构 — 三通道防御

```
┌─────────────────────────────────────────────────────────────────┐
│                     三通道防御架构                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  通道一·硬约束（结构层）                                          │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Agent 只能调用 MCP 工具白名单                               │   │
│  │ • 无 SQL/API/DB 直接访问权                                 │   │
│  │ • 无文件系统访问权                                         │   │
│  │ • 无网络出站权（除 MCP Server）                            │   │
│  │ • 工具白名单来自 Manifest 编译结果                          │   │
│  └──────────────────────────────────────────────────────────┘   │
│                          ▼                                        │
│  通道二·软约束（语义层）                                          │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 两步 MCP 工具，概念片段注入 LLM 上下文（见 §3.6）           │   │
│  │ • resolve_intent(query) → 精确概念引用 + alternatives     │   │
│  │ • query_ontology(context, concept) → 结构化定义           │   │
│  │ • LLM 对映射结果做 sanity check，再进入执行推理             │   │
│  │ • 按需查询，避免全量加载 Manifest                           │   │
│  └──────────────────────────────────────────────────────────┘   │
│                          ▼                                        │
│  通道三·校验层（执行前）                                          │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Agent 输出经四重校验后才能执行                              │   │
│  │ ① Schema 白名单：实体/属性/枚举值必须在 Manifest 中定义    │   │
│  │ ② 行为前置条件：状态机校验 + 校验规则（如物料齐套）        │   │
│  │ ③ 权限矩阵：角色×对象×字段 三级权限校验                    │   │
│  │ ④ 聚合根入口：不能绕过聚合根直接操作子对象                 │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 7. 风险评估与缓解

### 7.1 风险矩阵

| # | 风险 | 类型 | 概率 | 影响 | 等级 |
|---|------|------|------|------|------|
| R1 | 本体建模门槛高，用户上手困难 | 产品 | 高 | 高 | 🔴 |
| R2 | Agent 三通道防御存在绕过漏洞 | 安全 | 中 | 极高 | 🔴 |
| R3 | 制造域行业知识不足，模型不贴合实际 | 领域 | 中 | 高 | 🟡 |
| R4 | MCP 协议演进导致兼容性问题 | 技术 | 中 | 中 | 🟡 |
| R5 | Palantir 或国内大厂推出竞争产品 | 市场 | 中 | 高 | 🟡 |
| R6 | 企业决策周期长，销售转化慢 | 商业 | 高 | 中 | 🟡 |
| R7 | 团队 DDD/事件驱动架构能力不足 | 执行 | 低 | 高 | 🟢 |
| R8 | 数据源连接信息泄露 | 安全 | 低 | 极高 | 🟢 |

### 7.2 缓解策略

#### R1：建模门槛高
- **缓解**：预置制造域参考模型（附录 C）；可视化拖拽建模；数据库逆向生成骨架；AI 辅助建模（V2.0）
- **指标**：新用户 2 小时内完成第一个聚合根定义

#### R2：三通道防御绕过
- **缓解**：独立安全审计（每版本）；Bug Bounty 计划；沙箱采用进程级隔离（非容器级）
- **指标**：安全审计零高危漏洞通过

#### R3：领域知识不足
- **缓解**：与制造域 SME 联合设计参考模型；种子客户深度共建；开放模板贡献机制
- **指标**：3 家种子客户生产环境验证通过

#### R4：MCP 协议兼容性
- **缓解**：MCP Server 独立部署，协议适配层隔离变化；支持多协议版本并存
- **指标**：MCP 协议升级后 1 个月内完成适配

#### R5：竞品压力
- **缓解**：聚焦差异化（三通道防御 + 私有化 + 国产化）；快速占领制造域标杆客户形成壁垒
- **指标**：Year 1 内至少 1 家行业头部客户案例对外发布

#### R6：销售周期长
- **缓解**：免费社区版降低试用门槛；POC 缩至 1 周（预置模板）；与 AI 平台联合销售
- **指标**：POC → 付费转化率 >25%

#### R7：团队能力
- **缓解**：核心架构师具备 DDD 实战经验；引入外部 DDD 顾问；内部知识沉淀（ADRs + 技术文档）
- **指标**：核心模块代码审查覆盖率 100%

#### R8：数据源凭证泄露
- **缓解**：凭证加密存储（KMS）；Manifest 不含明文凭证（仅引用凭证 ID）；访问日志告警
- **指标**：安全扫描零明文凭证检出
