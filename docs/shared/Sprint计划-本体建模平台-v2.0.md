## 1. 总体估算

| 指标 | 数值 |
|------|------|
| 总 Story 数 | 32 |
| MVP (P0) Story 数 | 17 |
| 增强 (P1) Story 数 | 10 |
| 远期 (P2) Story 数 | 3 |
| 估算总 Sprint 数 | **10 个 Sprint**（2 周/Sprint） |
| MVP 交付 Sprint | Sprint 6 末（约 12 周） |
| V1.5 交付 Sprint | Sprint 8 末（约 16 周） |
| V2.0 交付 Sprint | Sprint 10 末（约 20 周） |

**故事点估算规则**：
- 1 点 = 约 1 人天（含前后端 + 测试）
- 团队规模假设：4–6 人（2 后端 + 1 前端 + 1 全栈 + 1 QA + 1 PM）
- 每 Sprint 容量：约 20–25 点

## 2. Sprint 详细拆分

### Sprint 1：地基 — 语义层核心（18 点）

**目标**：限界上下文 + 聚合根 + 对象属性 + 关系类型 + 数据源配置 全部可用。

| ID | Story | 点 | 依赖 |
|----|-------|----|------|
| **US-S01** | 创建限界上下文并绑定本体 | 5 | — |
| **US-S03** | 定义聚合根 | 5 | S01 |
| **US-S04** | 定义对象类型与属性 | 5 | S03 |
| **US-S07** | 定义关系类型 | 3 | S04 |

**交付物**：
- 限界上下文 CRUD API
- 聚合根创建 + 子对象挂载
- 属性定义（含类型校验）+ 拖拽排序
- 关系定义（组合/聚合/引用/依赖 + 基数）

**注意**：S02 (业务场景) 虽然 P1，但架构上 S04 的 `business_scenarios` 表已建好，Sprint 4 补前端即可。

---

### Sprint 2：地基 — 数据源 + 治理基础（20 点）

**目标**：数据源配置可用 + 角色权限体系就位 + 沙箱框架就位。

| ID | Story | 点 | 依赖 |
|----|-------|----|------|
| **US-S08** | 配置数据源与获取方式 | 5 | S04 |
| **US-G01** | 定义角色与对象级权限 | 8 | S04 |
| **US-G04** | 配置 AI Agent 沙箱 | 5 | G01 |
| 基础框架 | DB 迁移、API 框架、认证 | 2 | — |

**交付物**：
- SQL/API/MCP 三种数据源类型配置
- 角色 CRUD + 对象级权限矩阵（READ/WRITE/DELETE/EXECUTE）
- 沙箱配置（工具白名单 + 聚合根白名单 + 行为白名单 + 频率限制）
- Flyway V2 迁移执行 + API 认证中间件

---

### Sprint 3：行为层 MVP（22 点）

**目标**：行为定义 + 校验规则 + 指标定义全部可用。

| ID | Story | 点 | 依赖 |
|----|-------|----|------|
| **US-B01** | 定义行为（含调用模式） | 8 | S03 |
| **US-B03** | 定义校验规则并关联行为 | 8 | B01 |
| **US-B05** | 定义业务指标 | 5 | S04 |
| 基础 | 行为引擎骨架（执行管线） | 1 | B01 |

**交付物**：
- 行为 CRUD（名称/聚合根/调用模式/输入参数/声明式事件发布）

- 指标 CRUD（公式/数据源/维度/聚合周期）
- 行为执行管线接口（为 A05 预留）

---

### Sprint 4：事件层 MVP + 场景（22 点）

**目标**：领域事件 + 事件路由 + 事件处理器矩阵 + 业务场景 全部可用。

| ID | Story | 点 | 依赖 |
|----|-------|----|------|
| **US-E01** | 定义领域事件 | 5 | B01 |
| **US-E03** | 定义事件路由 | 5 | E01 |
| **US-E04** | 定义事件处理器矩阵 | 8 | E01, S02, S06 (状态需先有) |
| **US-S02** | 在上下文中定义业务场景 | 3 | S01 |
| **US-S06** | 定义对象状态与状态机 | 5 | S04 |

**注意**：S06 虽然 P1，但 E04 依赖状态机维度的 `required_state_id`。Sprint 4 至少实现状态/转换的基本定义（不包含可视化），可视化 AC-4 推迟到 Sprint 7。

**交付物**：
- 领域事件 CRUD + Payload Schema 定义
- 事件路由配置（CONTEXT / EXTERNAL_SYSTEM + 条件过滤）
- 事件处理器矩阵（事件×场景×状态→处理行为）
- 业务场景 CRUD
- 状态机（状态节点 + 转换 + 驱动行为关联）

---

### Sprint 5：工作流 + Manifest 编译器（20 点）

**目标**：建模工作流闭环 + Manifest 一键编译。

| ID | Story | 点 | 依赖 |
|----|-------|----|------|
| **US-G05** | 建模工作流与审批 | 10 | S01–S04 全部 |
| **US-A01** | 编译本体为 Agent Manifest | 8 | 全部语义/行为/事件/治理层 |
| 基础 | Manifest 编译器引擎 | 2 | A01 |

**交付物**：
- 草稿→审核→发布三阶段工作流（状态机 + 只读锁定 + 审批链）
- 审核批注（通过/需修改/驳回）+ 打回重审
- 批准发布自动触发 Manifest 编译
- Manifest JSON 输出（七段元数据完整）+ 编译错误结构化报告
- Manifest 版本历史存储

---

### Sprint 6：MCP 消费 + 三通道防御 + MVP 集成测试（24 点）

**目标**：Agent 经三步 MCP 工具完成查询与执行，三通道防御端到端生效。**MVP 里程碑交付。**

| ID | Story | 点 | 依赖 |
|----|-------|----|------|
| **US-A03** | Agent 经 MCP 解析意图并查询本体 | 9 | A01 |
| **US-A05** | Agent 执行行为（三通道防御） | 10 | A03, G04, B03 |
| 集成 | MCP Server 独立服务搭建 | 2 | — |
| 集成 | MVP 端到端集成测试（含 alternatives 纠错路径） | 3 | A05 |

**交付物**：
- MCP Server 独立进程（Port 8090）启动 + 加载 Manifest
- `resolve_intent` 工具：自然语言 → `IntentResolution`（含 `alternatives`）
- `query_ontology` 工具：精确概念引用 → 完整概念片段
- `execute_action` 工具：统一执行入口（三通道完整链路；MVP 不用每行为独立 Tool）
- 通道一：MCP 工具白名单拦截
- 通道二：本体概念片段注入 LLM 上下文
- 通道三：Schema + 规则 + 权限 + 聚合根入口四重校验
- 制造域 MVP 场景端到端验收（生产订单完整生命周期）

**🏁 MVP 里程碑**：Sprint 6 结束时交付 MVP（17 个 P0 Story 全部可用）。

## 4. 风险与缓冲

| 风险 | 影响 Sprint | 缓解 | 缓冲 |
|------|-----------|------|------|
| Manifest 编译器实现比预期复杂 | Sprint 5 | 先输出最小可用 Manifest（仅语义+行为段），其余段 Sprint 6 补 | +1 Sprint |
| MCP 协议细节打磨耗时 | Sprint 6 | Sprint 5 末开始 MCP Server 骨架搭建 | +0.5 Sprint |
| 校验规则表达式引擎（JSONB DSL）设计迭代 | Sprint 3 | Sprint 3 先支持简单规则（字段比较），复杂表达式 Sprint 7 增强 | 内化 |
| 三通道防御集成调试 | Sprint 6 | 每个通道独立单元测试 + Sprint 5 末开始集成 | +1 Sprint |
| 前端可视化（状态机图/血缘 DAG）工作量低估 | Sprint 7, 9 | 优先表格视图，图形视图降级为 P2 | 内化 |

**总缓冲**：2 Sprint（约 4 周）。最坏情况 MVP 延迟到 Sprint 7–8 交付。
