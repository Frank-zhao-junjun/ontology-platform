## 1. 产品概述

### 1.1 产品定位

**本体建模即服务**——企业AI平台的"业务语义中台"

本体建模平台是连接企业业务数据与大模型智能能力的核心中间层。它将分散在ERP、CRM、MES等系统中的业务数据，转化为AI可理解、可执行、可审计的业务语义模型，从而解决大模型"不懂业务、幻觉频发"的企业级应用困境。

#### 1.1.1 Non-Goals —— 本平台明确不做什么

PRD 的范围需要有明确的边界。以下是本平台**明确不纳入**的领域，防止需求膨胀和团队理解偏差：

| 不做 | 原因 | 替代方案 |
|------|------|---------|
| **不存储实例数据** | 平台是语义中台，不是数据中台——只记录"从哪里获取"的元数据 | 消费方（AI Agent / 应用）自行调用数据源获取实例数据 |
| **不做知识图谱推理** | 本体是 Schema 层，图谱推理是下游消费者的能力 | 知识图谱平台读取本体 Schema 后自行实现推理 |
| **不做 ETL / 数据同步** | 平台不执行数据拉取、清洗或定时同步任务 | 数据中台 / CDC 工具（Debezium, Flink CDC）负责 |
| **不做 BI 报表** | 指标定义归本体，可视化渲染归 BI 工具 | BI 工具（Superset, Metabase, PowerBI）读取指标定义自行渲染 |
| **不做 LLM 训练 / Fine-tuning** | 平台提供语义上下文用于 RAG，不训练模型参数 | AI 平台（Dify, LangChain, Coze）负责模型层 |
| **不做工作流引擎** | 行为编排在 Agent 层完成，平台只管概念定义 | 工作流引擎（Temporal, Airflow）编排调用本体行为 |
| **不做 API Gateway** | 平台暴露本体 API，但路由/限流/认证由网关统一管理 | 企业 API Gateway（Kong, APISIX, Nginx） |
| **不做低代码/无代码应用搭建** | 本体是语义模型，不是应用开发平台 | 低代码平台（明道云, 简道云, Retool） |

**设计原则**：每一条 Non-Goal 都是对"为什么这个功能不在 PRD 里"的提前回答。当有人问"本体平台能不能做 XX"，先查这张表——如果不在表里，再评估是否放入下一版路线图。

### 1.2 核心价值主张

本体建模平台采用**四层架构**，整合 DDD（领域驱动设计）与事件驱动架构：

| 层级 | 定位 | 核心内容 |
|-----|------|---------|
| **语义层**（Semantic Layer） | 业务「名词」— 定义业务是什么 + 记录数据获取方法（仅记录，不执行获取） | 限界上下文、对象类型、属性、关系、聚合（聚合根）、值对象、接口抽象；数据源、数据获取方式（SQL/API/MCP）、数据血缘 |
| **动力层**（Kinetic Layer） | 业务「动词」— 定义能做什么 + 如何衡量 | 行为、校验规则、副作用配置、事务边界；指标定义 |
| **事件层**（Event Layer） | 业务「信号」— 定义发生了什么 | 领域事件、集成事件、事件处理器、事件存储、事件总线 |
| **治理层**（Governance Layer） | 业务「权限」— 定义谁能做什么 | 角色、对象级权限、字段级权限、条件权限、AI Agent 沙箱 |

### 1.4 目标市场

#### 1.4.1 目标客户画像

| 维度 | 描述 |
|-----|------|
| **行业** | 金融、制造、政务、能源、医疗、零售等需要复杂业务管控的行业 |
| **企业规模** | 中大型企业（年营收5亿以上），已具备一定数字化基础 |
| **部门** | 科技/信息部门主导，数字化转型办公室/数据中台团队协同 |
| **决策链** | CTO/CIO → 架构师 → 业务部门负责人 |
| **关键决策人** | 对AI落地有明确诉求，已尝试过通用大模型但效果不佳的企业 |

#### 1.4.2 典型场景

```
场景1：智能客服升级
─────────────────────────────────
现状：客服机器人答非所问，无法处理复杂业务咨询
本体建模：定义"产品-规格-价格-库存-政策"语义关系
效果：AI可准确回答库存查询、订单状态、退换货政策

场景2：智能决策辅助
─────────────────────────────────
现状：领导需要看报表才能做决策，数据分析周期长
本体建模：构建"订单-客户-产品-供应链"决策知识图
效果：AI可实时响应"大客户订单延误会有什么影响"

场景3：流程自动化
─────────────────────────────────
现状：RPA机器人执行固定规则，异常处理需人工介入
本体建模：定义"工单-审批-执行-反馈"业务行为语义
效果：AI可处理半结构化业务，根据语义理解选择正确流程
```

### 1.5 与现有AI平台的集成能力

本体建模平台支持与主流AI平台的集成：

| 平台类型 | 集成方式 | 价值 |
|---------|---------|------|
| **Dify** | MCP协议 / API | 为Dify应用提供结构化业务上下文，减少LLM幻觉 |
| **Coze/扣子** | MCP协议 / 插件 | 为Bot提供业务实体查询和行为执行能力 |
| **LangChain** | Tool/Retriever | 作为企业知识库的语义层，增强RAG效果 |
| **自建AI平台** | SDK / API | 提供统一的企业语义API，支持多模型调用 |
| **国产大模型** | MCP适配层 | 支持百度文心、阿里通义、智谱GLM等国产模型 |

### 1.6 产品成功指标

以下量化 KPI 定义了平台在 MVP、V1.5 和 V2.0 三个阶段的成功标准。所有指标均基于可测量的数据，避免模糊目标。

#### 1.6.1 用户与增长指标

| 指标 | 定义 | MVP 目标 | V1.5 目标 | V2.0 目标 |
|------|------|---------|----------|----------|
| **月活建模用户（MAU）** | 每月至少完成一次本体编辑操作的去重用户数 | 20 | 100 | 500 |
| **周活跃率** | WAU / MAU，衡量用户粘性 | >40% | >55% | >60% |
| **本体创建数（月）** | 每月新创建的本体数量 | 30 | 150 | 600 |
| **租户数** | 付费 + 试用 + 免费租户总数 | 10 | 50 | 200 |

#### 1.6.2 效率与价值指标

| 指标 | 定义 | MVP 目标 | V1.5 目标 | V2.0 目标 |
|------|------|---------|----------|----------|
| **本体建模耗时降低** | 相较于手写 Prompt/代码定义业务语义，建模耗时减少比例 | 降低 60% | 降低 75% | 降低 85% |
| **AI 回答准确率提升** | 有本体语义上下文 vs 无本体，AI 回答业务的准确率差值 | 提升 20% | 提升 35% | 提升 50% |
| **幻觉率降低** | 有本体约束时 AI 产生幻觉（捏造字段/概念）的频率降低比例 | 降低 50% | 降低 70% | 降低 85% |
| **本体复用率** | 已发布本体中被 2+ Agent 消费的比例 | 30% | 60% | 80% |

#### 1.6.3 质量与满意度指标

| 指标 | 定义 | MVP 目标 | V1.5 目标 | V2.0 目标 |
|------|------|---------|----------|----------|
| **NPS（净推荐值）** | 用户推荐意愿评分 | >30 | >45 | >55 |
| **API 可用性** | 本体平台 API 的月度 uptime | >99.5% | >99.9% | >99.95% |
| **POC → 付费转化率** | 完成概念验证后转为付费客户的比例 | >25% | >35% | >45% |
| **用户自助率** | 不使用人工支持即可完成建模操作的用户占比 | >60% | >80% | >90% |

**数据来源**：
- 用户行为指标 → 平台内埋点 + Mixpanel / 百度统计
- AI 准确率/幻觉率 → 定期 A/B 评测（有本体 vs 无本体的对照实验）
- NPS → 产品内 NPS 弹窗问卷（每季度）
- API 可用性 → Prometheus + Grafana 监控面板

## 2. 市场分析

### 2.1 痛点分析

#### 2.1.1 企业AI应用的四大痛点

```
┌────────────────────────────────────────────────────────────────────┐
│                      企业AI应用四大痛点                            │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  痛点1：语义鸿沟 (Semantic Gap)                                    │
│  ─────────────────────────────────────────────────────────────   │
│  • 业务系统用"客户编码"标识客户，AI不知道这对应"客户"实体        │
│  • 字段"status=3"对AI毫无意义，需要业务字典翻译                  │
│  • 不同系统同一概念口径不一致，如"活跃用户"在不同系统定义不同     │
│                                                                    │
│  痛点2：幻觉失控 (Hallucination)                                   │
│  ─────────────────────────────────────────────────────────────   │
│  • AI可能生成不存在的订单编号                                     │
│  • AI可能承诺不存在的库存                                         │
│  • AI可能提供违背业务政策的建议                                   │
│                                                                    │
│  痛点3：执行闭环缺失 (Execution Gap)                              │
│  ─────────────────────────────────────────────────────────────   │
│  • AI只能给建议，无法执行操作                                     │
│  • 需要执行时仍需人工介入                                         │
│  • 缺乏对执行结果的追踪和反馈                                     │
│                                                                    │
│  痛点4：知识流失 (Knowledge Loss)                                 │
│  ─────────────────────────────────────────────────────────────   │
│  • 业务专家的经验无法显式化                                       │
│  • 跨部门协作缺乏统一语义                                         │
│  • 人员变动导致知识断层                                           │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

#### 2.1.2 痛点量化分析

| 痛点 | 表现形式 | 影响 | 量化指标 |
|-----|---------|------|---------|
| 语义鸿沟 | AI理解偏差 | 回答准确率低、用户信任度下降 | 客服满意度<60% |
| 幻觉失控 | 生成错误信息 | 业务风险、客诉增加 | 业务错误率>5% |
| 执行闭环缺失 | 需人工处理 | 效率低、成本高 | 人工介入率>30% |
| 知识流失 | 经验难传承 | 培训成本高、一致性差 | 新人上手周期>3个月 |

#### 2.2.2 Palantir Foundry深度分析

**核心优势**：
- 四层架构（语义层/动力层/事件层/治理层）设计成熟
- 与AI Agent深度整合，决策-执行闭环完整
- 全球头部客户验证，案例丰富

**主要局限**：
- 价格高昂，仅适合超大型企业
- 纯SaaS模式，不支持私有化部署
- 对国产大模型和国内企业系统集成支持弱
- 实施周期长，本地化服务能力有限

#### 2.2.3 自定义开发方案分析

**优势**：
- 完全可控，可根据企业需求定制
- 可以复用现有技术栈

**劣势**：
- 需要从零开始构建本体建模能力
- 技术风险高，可能需要1-2年才能稳定
- 需要专职团队维护
- 难以沉淀为可复用产品

### 3.2 核心功能模块详解

#### 3.2.1 语义层（Semantic Layer）—— 定义业务「名词」+ 记录数据获取方法 [MVP→V1.5]

**功能描述**：将企业业务实体抽象为本体对象，建立属性、关系和聚合体系。

| 功能 | 描述 | 优先级 | 阶段 |
|-----|------|-------|------|
| **限界上下文** | **本体与限界上下文 1:1 严格绑定**——每个本体对应一个限界上下文，本体的边界就是语言的边界。同一个术语（如「客户」）在不同上下文中可以有不同含义，由各自的本体独立定义 | P0  | ✅ MVP |
| **对象类型定义** | 定义业务实体类型（如客户、订单、产品、设备），支持继承和接口 | P0  | ✅ MVP |
| **属性建模** | 为对象定义属性（名称、类型、来源映射），支持计算属性 | P0  | ✅ MVP |
| **关系类型定义** | 定义对象间关系（1:1、1:N、M:N），支持关系属性 | P0  | ✅ MVP |
| **聚合(聚合根)** | 定义以聚合根为核心的紧密关联对象集群，建立事务一致性边界 | P1  | V1.5 |
| **值对象** | 定义无独立标识的概念性对象（如地址、金额、日期范围），可嵌入实体 | P1  | V1.5 |
| **接口抽象** | 定义可被多对象类型实现的公共接口（如「可审批对象」） | P1  | V1.5 |
| **数据映射** | 将本体对象属性映射到数据源的获取方式。**平台不存储实例数据、不执行数据获取**——只记录"从哪里获取、怎么获取"的方法元数据。支持三种接入方式：SQL直查、API调用、MCP工具调用 | P0  | ✅ MVP |
| **数据源** | 定义外部数据系统（ERP/OData/CRM/MES等），包含连接配置、认证方式和协议类型 | P0  | ✅ MVP |
| **数据血缘** | 记录每条数据字段的完整来源链路——从哪个数据源、哪个接口、经过什么转换而来，AI回答"这个数据哪来的"时自动引用 | P1  | V1.5 |
| **本体导入导出** | 支持RDF/OWL格式导入，预置行业本体模板 | P1  | V1.5 |

**上下文映射（Context Mapping）**：当多个本体/限界上下文需要协作时，通过以下模式建立上下文间的翻译关系：

| 映射模式 | 描述 | 适用场景 |
|---------|------|---------|
| **防腐层（ACL）** | 在本体A和本体B之间建立翻译层，将外部概念转换为本地语义 | 集成外部系统（如ERP→本体）时，保护本体内部模型不受外部模型污染 |
| **共享内核（Shared Kernel）** | 两个本体共享一部分共同的概念定义（如「员工ID」、「部门编号」） | 两个紧密协作的上下文需要共享基础数据定义 |
| **开放主机（Open Host）** | 本体对外暴露一套稳定的集成事件协议，外部系统通过此协议消费 | 本体作为企业内部核心语义源，需要对外提供一致的数据视图 |
| **上下游（Upstream/Downstream）** | 上游本体主导模型定义，下游本体跟随（如「订单」上下文定义订单模型，「库存」上下文按此模型消费事件） | 事件驱动集成场景，事件发布方是上游 |

**DDD 关键原则**：
- **一个大模型统治所有是万恶之源**——不要试图用一个本体覆盖整个企业的所有概念
- **每个限界上下文拥有自己的通用语言（Ubiquitous Language）**——「客户」在CRM上下文中是"潜在买家"，在ERP上下文中是"已签约甲方"
- **上下文映射是本体之间唯一的耦合点**——本体内部高度内聚，跨本体通过映射模式解耦

**数据接入方式（Data Access Methods）**：语义层**仅记录元数据**——存储"从哪里获取数据、怎么获取"的方法描述，**平台自身不执行数据获取**。建模时定义数据源和获取方式，运行时由消费方（AI Agent / 应用）读取这些元数据，自行调用正确的接口获取实例数据。

数据接入方式分为三种，覆盖企业全部数据获取场景：

| 接入方式 | 适用场景 | 配置内容 | 示例 |
|---------|---------|---------|------|
| **SQL 直查** | 数据库/数据仓库 | JDBC连接串、SQL模板、参数绑定 | `SELECT * FROM orders WHERE id = :orderId` |
| **API 调用** | REST/OData/SOAP | 端点URL、HTTP方法、请求模板、响应路径映射 | `GET /sap/opu/odata/sap/API_SALES_ORDER_SRV/A_SalesOrder('{id}')` |
| **MCP 工具调用** | AI Agent 工具链 | MCP Server名称、工具名称、参数Schema | `mcp__sap_odata__get_sales_order({id})` |

**数据映射示例——三种接入方式对比**

同一个「订单金额」属性，三种不同的获取方式：

```
属性：订单金额 (total_amount)

方式1 — SQL 直查:
  ├─ 数据源：ERP_PostgreSQL
  ├─ 查询：SELECT total_amount FROM orders WHERE order_id = :orderId
  └─ 类型：DECIMAL → 返回值直接填充属性

方式2 — API 调用:
  ├─ 数据源：SAP S/4HANA Cloud
  ├─ 端点：GET /sap/opu/odata/sap/API_SALES_ORDER_SRV/A_SalesOrder('{orderId}')
  ├─ 认证：OAuth 2.0 (Client Credentials)
  ├─ 响应路径映射：$.d.TotalNetAmount → total_amount
  └─ 缓存策略：TTL 5分钟（订单金额变更不频繁）

方式3 — MCP 工具调用:
  ├─ MCP Server：sap-odata-server
  ├─ 工具名称：get_sales_order
  ├─ 参数映射：{ "orderId": "{{object.order_id}}" }
  ├─ 响应路径映射：$.TotalNetAmount → total_amount
  └─ 回退策略：MCP 不可用时自动降级到 API 调用
```

#### 3.2.2 动力层（Kinetic Layer）—— 定义业务「动词」[MVP→V1.5]

**功能描述**：定义对业务对象的可执行行为，将 AI 建议转化为可治理的写操作。行为是动力层的**唯一核心元语**。

**核心约束——聚合根入口规则**：外部只能通过聚合根访问聚合内部实体。行为挂载到聚合根时标记为「聚合入口行为」，直接挂载到非聚合根对象时系统给出警告（聚合内部一致性无法保证）。这是 AI 安全执行边界的第一道防线。

| 功能 | 描述 | 优先级 | 阶段 |
|-----|------|-------|------|
| **行为定义** | 定义业务行为（如审批订单、取消订单、分配任务），包含参数、返回值和入口约束。行为必须指定挂载目标——挂载到聚合根时自动标记为「聚合入口行为」，受聚合事务边界保护；挂载到非聚合根对象时系统提示警告 | P0  | ✅ MVP |
| **校验规则** | 定义行为执行前后的业务规则校验 | P0  | ✅ MVP |
| **副作用配置** | 配置行为触发的关联操作（通知、回写、事件发布） | P1  | V1.5 |
| **事务边界** | 定义聚合行为的原子性事务范围，确保聚合内数据一致性 | P1  | V1.5 |
| **指标定义** | 定义业务指标（如「月度销售额」「客户流失率」）。指标是一种特殊的查询行为——定义计算公式、维度拆分和聚合逻辑。AI 做业务分析时自动引用指标，回答「上个月销售额多少」→ 直接调用预定义指标计算 | P1  | V1.5 |

#### 3.2.3 事件层（Event Layer）—— 定义业务「信号」★新增 [V1.5→V2.0]

**功能描述**：定义行为执行后产生的业务事件，实现聚合间的松耦合通信和跨系统集成。**事件由行为主动声明**——行为定义中的 `events` 数组声明了「执行完成后发布哪些事件」，系统自动生成并发布，不依赖建模者事后手动配置。

| 功能 | 描述 | 优先级 | 阶段 |
|-----|------|-------|------|
| **领域事件** | 定义聚合内行为完成后产生的业务事实（如 OrderApproved、InventoryDeducted）。事件由行为主动声明——行为定义中配置 `events` 数组，系统在执行完成后自动发布。包含事件ID、时间戳、聚合ID和事件数据 | P1  | V1.5 |
| **集成事件** | 定义跨限界上下文/跨系统的外部事件，用于系统间异步通信（如 ERP 订单同步到 WMS） | P1  | V1.5 |
| **事件处理器** | 定义事件触发后的处理逻辑——可以是新行为、通知、Webhook 回调或外部 API 调用 | P1  | V1.5 |
| **事件存储** | 持久化所有事件（Event Store），支持事件溯源和审计回放。P2阶段将在此基础上引入 CQRS 读模型投影 | P2  | V2.0 |
| **事件总线** | 事件发布/订阅基础设施，支持内存队列（开发）和 Kafka/RabbitMQ（生产）两种模式 | P1  | V1.5 |

**架构预留——CQRS 读模型投影**（P2）：

当前 MVP 阶段行为统一处理读写（查询行为直接读取对象属性/关系图）。P2 阶段将在事件存储基础上引入 CQRS：
- **写模型**：行为通过聚合根修改数据，产生领域事件
- **读模型投影（Projection）**：事件处理器订阅领域事件，实时更新独立投影表，支持跨聚合复杂查询
- **最终一致性**：写模型完成后，投影异步更新（延迟通常在毫秒级）

**DDD 映射**：

| 事件层要素 | DDD 对应 | 说明 |
|-----------|---------|------|
| 领域事件 | Domain Event | 聚合内部的事实记录，用过去式命名（如 OrderSubmitted） |
| 集成事件 | Integration Event | 跨限界上下文的约定消息，通常是扁平化 DTO |
| 事件处理器 | Event Handler | 对事件的反应逻辑，可以是新行为的触发 |
| 事件存储 | Event Store | 不可变的事件序列，支持事件溯源 CQRS |

#### 3.2.4 治理层（Governance Layer）—— 定义「谁能做什么」[MVP]

**功能描述**：基于角色和属性的动态权限控制，确保 AI 操作在安全边界内执行。**权限挂在限界上下文上**——不同上下文可以有不同的角色定义和权限模型。

权限模型分为两层：
- **全局默认权限**：跨所有本体通用的基础角色（如「系统管理员」），定义在平台级别
- **上下文权限**：每个限界上下文定义自己的角色和权限（如「订单审批员」只存在于订单上下文中），以 context_id 关联

| 功能 | 描述 | 优先级 | 阶段 |
|-----|------|-------|------|
| **角色权限模型** | 基于RBAC的权限体系，定义角色和权限集合。支持全局默认角色和上下文角色两级 | P0  | ✅ MVP |
| **对象级权限** | 控制角色对特定对象实例的访问权限，按限界上下文隔离 | P0  | ✅ MVP |
| **字段级权限** | 控制角色对对象属性的访问和修改权限，按限界上下文隔离 | P1  | V1.5 |
| **条件权限** | 基于对象属性动态计算权限（如「部门经理只能看本部门数据」） | P1  | V1.5 |
| **AI Agent沙箱** | 为AI Agent定义操作边界——Agent 进入某个限界上下文时自动加载该上下文的沙箱规则 | P0  | ✅ MVP |

| **解耦** | 本体平台不绑定MCP协议，可独立演进 |

##### 4.2.2.1 Tools（工具）

MCP Server 暴露的工具分为两大类，两者职责截然不同：

| 类别 | 来源 | 数量 | 示例工具 |
|------|------|------|---------|
| **平台级基础工具**（固定） | 平台内置，所有本体通用 | 3 个 | `resolve_intent`、`validate_instruction`、`traverse_graph` |
| **本体级业务工具**（动态编译） | 由 Agent Manifest 根据本体定义动态编译生成，不同本体的工具列表不同 | N 个 | `execute_action("approve_order")`、`query_ontology("SalesOrder")` |

**关键设计约束**：
1. **平台级工具固定**：无论接入哪个本体，MCP Server 始终暴露这 3 个基础工具，处理意图识别、指令校验、图遍历等通用能力。
2. **本体级工具动态编译**：Agent 接入某个本体时，MCP Server 先读取该本体的 Manifest，将其中定义的 `actions[]` 和 `objects[]` 编译为对应的 MCP 工具签名，动态挂载到当前会话的工具列表上。
3. **不同本体工具列表不同**：`sales_ontology` 暴露 `execute_action("approve_order")`，`inventory_ontology` 暴露 `execute_action("adjust_stock")`，两者完全隔离，Agent 只能看到当前所连接本体的工具。

#### 4.3.4 多租户架构

本平台面向多企业/多团队场景，需要在不同租户之间提供安全、公平、可扩展的资源隔离。

##### 隔离策略

| 阶段 | 策略 | 说明 |
|------|------|------|
| **MVP** | 共享数据库 + `tenant_id` 行级隔离 | 所有租户数据在同一 PostgreSQL 实例中，通过 RLS（Row-Level Security）策略和 `WHERE tenant_id = current_tenant_id()` 实现逻辑隔离 |
| **V1.5** | 独立 Schema（大客户专用） | 对隔离要求高的租户分配独立 PostgreSQL Schema，自带 Schema 级权限边界 |
| **V2.0** | 独立数据库实例（合规场景） | 金融/政务等强合规场景支持租户专属数据库实例，物理级隔离 |

**MVP RLS 策略示例**：

```sql
-- 为每个租户启用行级安全策略
ALTER TABLE ontology ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_ontology_isolation ON ontology
    FOR ALL
    USING (tenant_id = current_setting('app.current_tenant_id')::UUID);

-- API 中间件在请求入口设置当前租户上下文
-- SET app.current_tenant_id = '<tenant-id-from-jwt>';
```

##### 租户配额（MVP 默认值）

| 资源 | 免费租户 | 基础版 | 专业版 | 企业版 |
|------|---------|-------|-------|-------|
| **本体数量** | 2 | 10 | 30 | 100 |
| **对象类型（每本体）** | 20 | 100 | 500 | 2000 |
| **属性（每对象类型）** | 50 | 200 | 500 | 1000 |
| **数据源** | 3 | 10 | 50 | 200 |
| **MCP 工具调用（QPS）** | 100 | 500 | 2000 | 10000 |
| **API 请求（日）** | 10000 | 100000 | 1000000 | 10000000 |
| **存储（元数据）** | 500MB | 5GB | 50GB | 500GB |

##### 跨租户本体共享

- **模板市场**：已发布本体可选择"发布到模板市场"，其他租户可一键复制为独立副本
- **复制后独立演进**：复制后的本体与原模板完全解耦，各自独立版本管理，不共享更新
- **组织内共享**：同一组织（企业）下的多个租户可通过"组织模板库"共享本体，降低跨部门重复建模成本
- **V2.0 规划**：支持本体引用（Reference）而非复制——租户 B 可引用租户 A 的本体作为上游依赖，但需租户 A 授权

##### 租户管理 API

| 端点 | 说明 | 权限 |
|------|------|------|
| `POST /tenants` | 注册新租户 | 平台管理员 |
| `GET /tenants/{id}` | 查询租户信息及配额使用情况 | 租户管理员 |
| `PUT /tenants/{id}/quota` | 调整租户配额 | 平台管理员 |
| `GET /tenants/{id}/usage` | 获取租户资源用量报表 | 租户管理员 |

#### 4.4.2 本体版本化模型

本体的每次发布（`status` 从 `draft` → `published`）必须创建一份不可变快照，支持版本追溯、变更对比和回滚。

**版本快照表**（`ontology_version`）：

```sql
-- 本体版本快照（每次发布创建一条不可变记录）
CREATE TABLE ontology_version (
    id UUID PRIMARY KEY,
    ontology_id UUID REFERENCES ontology(id) NOT NULL,
    version VARCHAR(20) NOT NULL,              -- "v1.0", "v1.1", "v2.0"
    manifest_snapshot JSONB NOT NULL,            -- 完整 Manifest 快照（不可变）
    change_summary TEXT,                       -- 本次变更的摘要描述
    changelog JSONB,                          -- 变更明细 [{type, target, before, after, breaking}]
    is_breaking BOOLEAN DEFAULT FALSE,         -- 是否破坏性变更（影响下游消费方）
    migration_required BOOLEAN DEFAULT FALSE,  -- 是否需要数据迁移
    migration_script TEXT,                     -- 迁移脚本（如需）
    published_by UUID,
    published_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(ontology_id, version)              -- 同一本体版本号唯一
);

-- 版本快照索引
CREATE INDEX idx_version_ontology ON ontology_version(ontology_id, version);
CREATE INDEX idx_version_published ON ontology_version(published_at);
```

#### 4.4.3 变更日志模型

每次本体编辑操作（创建对象、修改属性、删除行为等）都记录一条变更事件，支持审计追溯和自动生成变更摘要。

```sql
-- 变更日志（每次编辑操作记录一条）
CREATE TABLE schema_change_log (
    id UUID PRIMARY KEY,
    ontology_id UUID REFERENCES ontology(id) NOT NULL,
    version_from VARCHAR(20),                   -- 变更前版本（编辑中可能为 NULL）
    version_to VARCHAR(20),                     -- 变更后版本（发布后填入）
    change_type VARCHAR(30) NOT NULL,          -- ADD_ENTITY | MODIFY_PROPERTY | DELETE_ACTION | CHANGE_RELATION | ...
    target_type VARCHAR(30) NOT NULL,          -- entity | property | action | event | permission | data_source
    target_name VARCHAR(200) NOT NULL,        -- 被修改对象的名称（如 "SalesOrder", "total_amount"）
    change_detail JSONB NOT NULL,             -- {before: {...}, after: {...}, diff: [...]}
    is_breaking BOOLEAN DEFAULT FALSE,        -- 是否破坏性变更
    migration_required BOOLEAN DEFAULT FALSE,
    migration_hint TEXT,                      -- 给下游消费方的迁移提示
    changed_by UUID,
    changed_at TIMESTAMP DEFAULT NOW()
);

-- 变更日志索引
CREATE INDEX idx_changelog_ontology ON schema_change_log(ontology_id, changed_at DESC);
CREATE INDEX idx_changelog_breaking ON schema_change_log(ontology_id, is_breaking) WHERE is_breaking = TRUE;
```

**变更类型枚举**（`change_type` 完整值集）：

| 类型 | 说明 | 常见场景 |
|------|------|---------|
| `ADD_ENTITY` | 新增对象类型 | 发布后新增"退货单"实体 |
| `MODIFY_ENTITY` | 修改对象类型（显示名、描述） | 重命名"客户"→"客户档案" |
| `DELETE_ENTITY` | 删除对象类型 | 发布后删除（需检查引用） |
| `ADD_PROPERTY` | 新增属性 | 新增"税率"属性 |
| `MODIFY_PROPERTY` | 修改属性（类型、映射、必填） | 修改"金额"类型为 Decimal |
| `DELETE_PROPERTY` | 删除属性 | 删除已废弃属性 |
| `ADD_RELATION` | 新增关系 | 订单→发票 关系 |
| `MODIFY_RELATION` | 修改关系 | 关系基数 1:N → M:N |
| `DELETE_RELATION` | 删除关系 | — |
| `ADD_ACTION` | 新增行为 | 新增"批量审批"行为 |
| `MODIFY_ACTION` | 修改行为（参数、前置条件） | 修改审批前置条件 |
| `DELETE_ACTION` | 删除行为 | — |
| `ADD_EVENT` | 新增事件 | 新增"库存预警"事件 |
| `MODIFY_PERMISSION` | 修改权限规则 | 收紧查询范围 |
| `ADD_DATA_SOURCE` | 新增数据源 | 接入新的 SAP 系统 |
| `MODIFY_DATA_SOURCE` | 修改数据源配置 | SAP 系统升级 URL 变更 |

**破坏性变更自动检测规则**：

| 变更操作 | 是否破坏性 | 检测逻辑 |
|-----------|-----------|---------|
| 删除实体/属性/行为 | ✅ 是 | 下游 Agent 引用此概念会报错 |
| 修改属性类型（String→Int） | ✅ 是 | 旧数据格式不兼容 |
| 修改行为参数（新增必填参数） | ✅ 是 | 调用方未传新参数会失败 |
| 修改枚举值（删除/重命名） | ✅ 是 | 下游引用旧枚举值失败 |
| 新增实体/属性/行为 | ❌ 否 | 下游不感知新增内容 |
| 修改显示名/描述 | ❌ 否 | 不影响机器解析 |
| 修改非必填参数为必填 | ✅ 是 | 向下不兼容 |

#### 4.5.5 可观测性方案

平台需要在生产环境中提供充分的运维可见性，确保问题可发现、可追溯、可恢复。以下按五大支柱设计：

##### Logging（日志）

- **格式**：结构化 JSON 日志，每条日志必须包含 `trace_id`、`ontology_id`、`tenant_id`、`user_id`
- **不记录**：请求 body 中的敏感字段（密码、Token、个人身份信息）
- **日志级别规范**：

| 级别 | 用途 | 示例 |
|------|------|------|
| ERROR | 需要人工介入的故障 | DB 连接池耗尽、MCP 工具执行异常 |
| WARN | 降级/重试/配额告警 | API 限流触发、租户配额 80% |
| INFO | 关键操作审计 | 本体发布、权限变更、Agent Manifest 编译 |
| DEBUG | 仅开发环境 | SQL 参数绑定、MCP 工具调用详情 |

- **采集**：Filebeat → Elasticsearch / OpenSearch → Kibana 查询面板

##### Metrics（指标）

Prometheus 采集以下黄金指标，Grafana 统一面板展示：

| 指标 | 类型 | 含义 | 告警阈值 |
|------|------|------|---------|
| `http_request_duration_seconds` (P50/P95/P99) | Histogram | API 延迟分布 | P99 > 200ms (警告), > 500ms (严重) |
| `http_requests_total` | Counter | 请求总量（按 endpoint + status 分组） | — |
| `http_errors_total` | Counter | 错误请求总数 | 错误率 > 1%（警告），> 5%（严重） |
| `mcp_tool_calls_total` | Counter | MCP 工具调用量（按工具名分组） | — |
| `mcp_tool_call_duration_seconds` | Histogram | MCP 工具执行耗时 | — |
| `db_connection_pool_active` | Gauge | 活跃数据库连接数 | > 80% 池容量（警告） |
| `ontology_compile_duration_seconds` | Histogram | Manifest 编译耗时 | P99 > 10s（警告） |
| `tenant_active_sessions` | Gauge | 当前活跃租户会话数 | — |

##### Tracing（分布式追踪）

- **协议**：OpenTelemetry（OTLP）
- **追踪链路**：MCP Client（Agent 平台）→ MCP Server → 本体平台 API → PostgreSQL / 图数据库
- **关键 Span**：
  - `mcp.resolve_intent` → 意图识别耗时
  - `ontology.query` → 本体查询耗时
  - `mcp.execute_action` → 行为执行 + 校验耗时
  - `manifest.compile` → Manifest 编译耗时
- **TraceID 传递**：从 MCP Client 的请求头 `X-Trace-ID` 透传至本体平台，缺失时在 MCP Server 入口自动生成
- **后端**：Jaeger / Grafana Tempo（与 Prometheus + Grafana 统一面板）

##### Health Check（健康检查）

| 端点 | 用途 | 检查项 | K8s Probe |
|------|------|-------|-----------|
| `GET /health` | 进程存活 | 返回 200 OK | Liveness Probe |
| `GET /ready` | 服务就绪 | DB 连接 + Redis 连接 + MCP 工具列表加载 | Readiness Probe |
| `GET /health/deep` | 深度检查 | 检查全部依赖（DB + Redis + 图数据库 + 数据源连通性抽样），仅内部运维使用 | — |

`/ready` 失败时 Kubernetes 自动从 Service 摘除该 Pod，防止流量打入未就绪实例。

##### Alerting（告警）

基于 Prometheus AlertManager，三级告警：

| 级别 | 触发条件 | 通知方式 |
|------|---------|---------|
| **P0-严重** | API 错误率 > 5% · DB 连接池耗尽 · 平台完全不可用 | 电话 + 企业微信 + 飞书 |
| **P1-警告** | API P99 > 500ms · 错误率 > 1% · 租户配额 95% · Pod 频繁重启 | 企业微信 + 飞书 |
| **P2-关注** | 磁盘使用 > 80% · 慢查询增多 · 认证失败率异常 | 日报邮件 |

### 4.8 本体发布与 Agent 合规机制

#### 4.8.1 核心问题

本体建模平台产出经过评审、正式发布的本体模型后，面临两个关键问题：

1. **交付形式**：本体模型以什么形式交付给 AI Agent？
2. **强制执行**：如何保证 Agent 100% 读取并遵守本体模型，而非绕过或忽略？

#### 4.8.2 核心原则：结构约束，而非意愿约束

Agent 遵守本体模型**不是靠"让它读"**，而是靠**让它别无选择**。不给 Agent 直接访问数据库、API、消息队列等基础设施的权限——所有操作必须经过本体平台暴露的 MCP 工具。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Agent 合规的核心逻辑                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ❌ 错误思路：                                                         │
│   ┌──────────────────────────────────────────────────────────────┐     │
│   │  在 System Prompt 里写 "请遵守本体模型" → Agent 可能忽略      │     │
│   └──────────────────────────────────────────────────────────────┘     │
│                                                                         │
│   ✅ 正确思路：                                                         │
│   ┌──────────────────────────────────────────────────────────────┐     │
│   │  不给 Agent 直接访问 SQL/API/DB 的权限                         │     │
│   │  Agent 能调用的工具只有本体平台 MCP 工具                       │     │
│   │  → 不是"愿不愿意遵守"，是"根本没有绕过的方法"                 │     │
│   └──────────────────────────────────────────────────────────────┘     │
│                                                                         │
│   对比：                                                                │
│   ┌────────────────────────────┬─────────────────────────────────┐     │
│   │  绕过本体（不可行）         │  走本体 MCP 通道（唯一通路）      │     │
│   ├────────────────────────────┼─────────────────────────────────┤     │
│   │  SELECT * FROM SNWD_SO     │  mcp__ontology__get_entity(      │     │
│   │  → DB连接不暴露给Agent      │    "SalesOrder", {id:"xxx"})    │     │
│   │                            │  → 过聚合根入口+权限校验          │     │
│   ├────────────────────────────┼─────────────────────────────────┤     │
│   │  PATCH /odata/.../SO(...)  │  mcp__ontology__execute_action(  │     │
│   │  → OData凭证不暴露          │    "approve_order", {...})       │     │
│   │                            │  → 过校验规则+事务边界            │     │
│   ├────────────────────────────┼─────────────────────────────────┤     │
│   │  直接写消息队列事件         │  mcp__ontology__publish_event(   │     │
│   │  → 队列访问权不给Agent      │    "OrderApproved", {...})       │     │
│   │                            │  → 过事件Schema校验+审计           │     │
│   └────────────────────────────┴─────────────────────────────────┘     │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 4.8.4 Agent Manifest 结构

发布后的本体模型编译为结构化的 **Agent Manifest**（JSON/YAML），版本化管理，每个发布版本不可变：

```yaml
manifest_version: "1.0"
ontology_id: "ont-sales-001"
ontology_version: "v2.3.0"          # 与本体版本严格对应
published_at: "2026-06-15T10:00:00Z"
publisher: "本体建模平台"

contexts:                            # 限界上下文（1:1 绑定本体）
  - id: "ctx-sales"
    name: "销售上下文"
    description: "管理从报价到发货的销售全流程"

entities:                            # 语义层：实体 + 属性 + 数据获取方式
  - id: "entity-sales-order"
    name: "SalesOrder"
    display_name: "销售订单"
    is_aggregate_root: true          # 聚合根标志
    context_id: "ctx-sales"
    properties:
      - name: "id"
        type: "string"
        required: true
      - name: "total_amount"
        type: "decimal"
        required: true
        access_method:               # 仅记录获取方式，平台不执行
          type: "odata"
          endpoint: "/A_SalesOrder('{id}')/TotalNetAmount"
          source: "sap_s4hana_prod"
      - name: "status"
        type: "enum"
        enum_values: ["DRAFT", "SUBMITTED", "APPROVED", "SHIPPED", "CANCELLED"]
    children:                         # 通过聚合根访问的子实体
      - entity_id: "entity-sales-order-item"

actions:                             # 动力层：行为 + 校验
  - id: "action-approve-order"
    name: "approve_order"
    display_name: "审批订单"
    target_entity: "entity-sales-order"
    is_aggregate_entry: true         # 强制通过聚合根入口
    parameters:
      - name: "order_id"
        type: "string"
        required: true
    validations:                     # 校验规则
      - rule: "status_must_be"
        value: "SUBMITTED"           # 当前状态必须为"已提交"
      - rule: "field_must_be"
        field: "total_amount"
        operator: "gt"
        value: 0                     # 金额必须大于0
    published_events:                # 声明式事件发布
      - event_id: "event-order-approved"

events:                              # 事件层：领域事件 + 集成事件
  - id: "event-order-approved"
    name: "OrderApproved"
    type: "domain_event"
    published_by_action_ids: ["action-approve-order"]
    payload_schema:
      order_id: "string"
      approved_by: "string"
      approved_at: "datetime"
```

##### 通道一：硬约束 — MCP 工具白名单（保证 100% 的核心）

这是强制执行的关键。MCP Server 将 Agent Manifest 中的实体、行为、事件编译为 MCP 工具列表，Agent 只能看到和调用这些工具：

| Manifest 要素 | 映射为 MCP 能力 | Agent 调用方式 |
|--------------|----------------|---------------|
| 实体（entity） | MCP Resource | `mcp__ontology__get_entity("SalesOrder", {id})` |
| 对象类型定义 | MCP Resource | `ontology://objects/SalesOrder` |
| 行为（action） | MCP Tool | `mcp__ontology__execute_action("approve_order", {...})` |
| 校验规则 | MCP Tool（dryRun） | `mcp__ontology__validate_instruction({...})` |
| 事件（event） | MCP Tool | `mcp__ontology__publish_event("OrderApproved", {...})` |
| 图遍历 | MCP Tool | `mcp__ontology__traverse_graph({...})` |
| 权限 | 工具内置校验 | 自动拦截越权操作 |

关键设计：**Agent 的 MCP 工具列表由 Manifest 动态生成，Agent 看不到任何数据库连接、API 密钥、消息队列地址。** 没有这些基础设施的访问权，Agent 在结构上就不可能绕过本体。

```json
// Agent 实际看到的 MCP 工具列表（示例）
{
  "tools": [
    {
      "name": "get_entity",
      "description": "获取业务实体。可用实体：SalesOrder, SalesOrderItem, Customer, Product",
      "inputSchema": {
        "type": "object",
        "properties": {
          "entity": { "type": "string", "enum": ["SalesOrder", "SalesOrderItem", "Customer", "Product"] },
          "id": { "type": "string" }
        },
        "required": ["entity", "id"]
      }
    },
    {
      "name": "execute_action",
      "description": "执行业务行为。可用行为：create_order, approve_order, cancel_order, create_customer",
      "inputSchema": {
        "type": "object",
        "properties": {
          "action": { "type": "string", "enum": ["create_order", "approve_order", "cancel_order", "create_customer"] },
          "parameters": { "type": "object" }
        },
        "required": ["action", "parameters"]
      }
    },
    {
      "name": "publish_event",
      "description": "发布业务事件。可用事件：OrderCreated, OrderApproved, OrderCancelled",
      "inputSchema": {
        "type": "object",
        "properties": {
          "event": { "type": "string", "enum": ["OrderCreated", "OrderApproved", "OrderCancelled"] },
          "payload": { "type": "object" }
        },
        "required": ["event", "payload"]
      }
    }
  ]
}
```

##### 通道二：软约束 — 运行时上下文注入

硬约束解决"能不能做"，软约束解决"知不知道"。**关键纠正：Manifest 片段不是会话启动时全局注入，而是运行时按需注入。**

实际请求链路（正确理解）：

```
用户: "帮我把订单 #123 的状态改成已发货"
        │
        ▼ ① 用户自然语言输入
┌───────────────────────────────────────────┐
│  LLM：过滤 + 意图识别                           │
│  → 判断用户意图是"修改订单状态"                │
│  → 不确定具体要用哪个行为/实体 → 调用 MCP 工具  │
└───────────────────────────────────────────┘
        │
        ▼ ② LLM 调用本体 MCP 工具查询
┌───────────────────────────────────────────┐
│  MCP: resolve_intent({                                │
│    query: "修改订单状态 已发货"                     │
│  })                                              │
│  → 本体返回：                                     │
│    "此操作对应行为: approve_order                    │
│     目标实体: SalesOrder (id=123)                 │
│     前置条件: status=SUBMITTED                     │
│     注意: 已发货(SHIPPED)不可取消"               │
└───────────────────────────────────────────┘
        │
        ▼ ③ 本体概念转化后注入上下文
┌───────────────────────────────────────────┐
│  LLM：将本体返回的概念片段注入当前上下文        │
│  注入内容（由本体 MCP 返回，非预设 System Prompt）: │
│    • 目标实体: SalesOrder                          │
│    • 可用行为: approve_order                       │
│    • 行为参数: order_id, approver, approve_note   │
│    • 前置条件: status == SUBMITTED                 │
│    • 转化后的准确任务：                             │
│      "调用 execute_action('approve_order',          │
│       {order_id:'123', approver:'user_X'})"       │
└───────────────────────────────────────────┘
        │
        ▼ ④ LLM 拆解准确任务
┌───────────────────────────────────────────┐
│  LLM：基于本体概念，拆解为精确工具调用参数      │
│  任务拆解:                                      │
│    1. 调用 validate_action 预校验               │
│    2. 调用 execute_action 正式执行              │
│    3. 检查返回 → 若成功 → 向用户汇报           │
└───────────────────────────────────────────┘
        │
        ▼ ⑤ 执行（进入硬约束通道一）
   [execute_action('approve_order', {...})]
   → 由 MCP Server 执行（通道一保障）
```

两种注入方式（均为运行时，非会话启动）：

| 注入方式 | 时机 | 内容来源 | 用途 |
|---------|------|---------|------|
| **MCP 工具返回注入** | LLM 调 MCP `resolve_intent` 后 | 本体 MCP 工具返回的概念片段（由本体平台生成） | 将用户模糊语言转化为精确的业务概念引用 |
| **RAG 按需检索** | Agent 需要进一步查详情时 | 特定实体/行为的完整定义（属性、校验规则、关系） | 让 Agent 了解"这个对象具体有什么字段" |

```yaml
# MCP 工具 resolve_intent 返回的概念片段（运行时生成，非预设）
# 这是注入到 LLM 上下文的实际内容示例

resolve_intent_result:
  intent_understood: "修改订单审批状态为已审批"
  matched_entities:
    - name: "SalesOrder"
      id: "123"
      current_status: "SUBMITTED"    # 当前状态，便于 LLM 判断前置条件
  matched_actions:
    - name: "approve_order"
      display_name: "审批订单"
      parameters:
        - name: "order_id"
          type: "string"
          required: true
        - name: "approver"
          type: "string"
          required: true
        - name: "approve_note"
          type: "string"
          required: false
      preconditions:
        - rule: "status_must_be"
          value: "SUBMITTED"
          current_status: "SUBMITTED"   # ✅ 满足前置条件
      side_effects:
        - type: "publish_event"
          event: "OrderApproved"
  warnings:
    - "注意：状态为 SHIPPED 的订单不可执行此操作"
  suggested_next_tool_call:
    tool: "execute_action"
    params:
      action: "approve_order"
      parameters:
        order_id: "123"
        approver: "（请补充当前用户标识）"
```

##### 通道三：校验层 — 输出守卫

这是最后一道防线。Agent 输出在真正执行前，经过本体平台的校验引擎：

```
Agent 输出 → 校验引擎 → 放行 / 拒绝

校验项：
┌──────────────────────────────────────────────────────┐
│ 1. 实体白名单校验                                      │
│    Agent 引用的实体名是否在 Manifest entities[] 中？   │
│    例："请查一下 OrderHeader" → ❌ 实体不存在          │
│                                                      │
│ 2. 属性白名单校验                                      │
│    Agent 引用的字段是否在对应实体的 properties[] 中？  │
│    例："订单的 priority 字段" → ❌ SalesOrder 无此属性 │
│                                                      │
│ 3. 枚举值白名单校验                                    │
│    状态/类型值是否在 enum_values[] 中？                 │
│    例："状态改为 SUPER_VIP" → ❌ 枚举值不存在          │
│                                                      │
│ 4. 行为前置条件校验                                    │
│    执行行为前，参数是否满足 validations[] 规则？        │
│    例：取消已发货订单 → ❌ 违反 status != SHIPPED      │
│                                                      │
│ 5. 权限校验                                           │
│    当前 Agent 角色是否有该行为/实体的执行权限？         │
│    例：客服角色审批订单 → ❌ 无权限                     │
│                                                      │
│ 6. 聚合根入口校验                                      │
│    是否尝试绕过聚合根直接访问子实体？                    │
│    例：直接操作 SalesOrderItem → ❌ 必须通过 SalesOrder│
└──────────────────────────────────────────────────────┘
```

校验失败时的响应格式：

```json
{
  "valid": false,
  "errors": [
    {
      "type": "enum_value_invalid",
      "entity": "SalesOrder",
      "field": "status",
      "received": "SUPER_VIP",
      "allowed": ["DRAFT", "SUBMITTED", "APPROVED", "SHIPPED", "CANCELLED"],
      "message": "status 的有效值为 DRAFT / SUBMITTED / APPROVED / SHIPPED / CANCELLED"
    }
  ],
  "suggestion": "请将 status 设为上述有效值之一后重试"
}
```

#### 4.8.6 三通道协同：纵深防御

三个通道不是"三选一"，而是**三层过滤器**，构成纵深防御体系。关键理解：**软约束（通道二）发生在硬约束（通道一）之前**——LLM 先通过软约束理解意图，再决定调哪个 MCP 工具；硬约束保证 LLM 只能调白名单工具；校验层最后兜底。

| 通道 | 触发时机 | 防御层面 | 保证力度 | 失效场景 | 被谁兜底 |
|------|---------|---------|---------|---------|---------|
| 软约束 | 意图识别阶段（调 MCP 前） | 认知层：LLM 知道什么 | 70-90%（依赖 LLM） | LLM 忽略/幻觉意图 | 校验层兜底 |
| 硬约束 | 工具选择阶段（调 MCP 时） | 入口层：LLM 能调什么 | **100%**（结构不可达） | 无 | — |
| 校验层 | 执行前（MCP Server 内部） | 出口层：参数是否合规 | **100%**（白名单拒绝） | 无 | — |

完整的 Agent 请求链路（正确理解）：

```
用户: "帮我把订单 #123 的状态改成已发货"
        │
        ▼ ① 用户自然语言输入
┌────────────────────────────────────────────┐
│  软约束（通道二）：意图识别 + 本体查询       │
│  LLM：过滤无关内容，识别意图 = "修改订单"   │
│  → 不确定对应哪个行为 → 调用 MCP 工具        │
│  → mcp__ontology__resolve_intent(               │
│       "修改订单状态 已发货")                 │
│  ← 本体返回：                               │
│     intent: "approve_order"                  │
│     entity: "SalesOrder"(id=123)            │
│     preconditions: [status==SUBMITTED]      │
│     → 此片段注入 LLM 上下文（运行时）       │
│  LLM：基于本体概念，拆解精确任务：           │
│     "应调用 execute_action('approve_order',   │
│      {order_id:'123'})"                     │
└────────────────────────────────────────────┘
        │
        ▼ ② LLM 决定调用哪个工具
┌────────────────────────────────────────────┐
│  硬约束（通道一）：MCP 工具白名单            │
│  LLM 只能看到本体暴露的 MCP 工具：          │
│  ✅ execute_action（白名单）                 │
│  ✅ validate_action（白名单）                │
│  ❌ UPDATE sales_order（不在白名单，不可见） │
│  → LLM 必须调用 execute_action              │
└────────────────────────────────────────────┘
        │
        ▼ ③ MCP Server 收到 execute_action 调用
┌────────────────────────────────────────────┐
│  校验层（通道三）：MCP Server 内部执行前     │
│  ✅ SalesOrder 实体存在                    │
│  ✅ approve_order 行为存在                 │
│  ✅ 当前 status=SUBMITTED ✓（满足前置条件） │
│  ✅ Agent 角色=sales_manager ✓（有权限）   │
│  ✅ 聚合根入口合规（通过 SalesOrder）        │
│  → 校验通过，放行执行                       │
└────────────────────────────────────────────┘
        │
        ▼ ④ 执行成功
    事件 OrderApproved 已发布 · 审计已记录 · 结果返回 LLM
        │
        ▼ ⑤ LLM 向用户汇报
    "订单 #123 已审批通过，状态变更为已审批。"
```

#### 4.8.7 设计总结

| 维度 | 设计决策 |
|------|---------|
| **交付形式** | 本体 → 编译为 Agent Manifest（版本化、不可变快照） |
| **分发渠道** | 三通道：MCP 工具白名单（硬）+ MCP 返回注入+RAG（软）+ 校验引擎（拦截） |
| **100% 保证** | 硬约束（结构上不可绕过）+ 校验层（白名单拒绝） |
| **软约束时机** | 运行时按需：用户自然语言 → LLM 意图识别 → 调 MCP `resolve_intent` → 本体返回概念片段注入上下文 → LLM 拆解精确任务 |
| **Agent 平台无关** | Manifest 是标准 JSON/YAML，MCP 是开放协议，适用于 Dify/Coze/LangChain 等所有主流平台 |
| **版本一致性** | Manifest 版本与本体版本严格对应，确保 Agent 使用的模型与平台定义一致 |

#### 6.3.2 客户生命周期价值（LTV）

```
LTV = ARPU × 客户生命周期 × 毛利率

假设：
- ARPU = ¥80,000/年（平均客单价）
- 客户生命周期 = 4年（企业客户平均）
- 毛利率 = 75%（SaaS业务特征）

LTV = ¥80,000 × 4 × 75% = ¥240,000

CAC（获客成本）控制目标：
- SMB: ¥20,000
- 中大企业: ¥50,000
- 大企业: ¥100,000

LTV/CAC 目标比率 > 3
```

## 8. 附录

### 8.1 术语表

| 术语 | 英文 | 定义 |
|-----|------|------|
| **本体** | Ontology | 企业业务概念的形式化表示，定义业务实体的类型、属性、关系和行为 |
| **限界上下文** | Bounded Context | 与本体 1:1 绑定——每个本体对应一个限界上下文，本体的边界就是语言的边界。同一个术语在不同上下文中可以有不同的含义 |
| **上下文映射** | Context Mapping | 多个限界上下文间的翻译关系定义，包含 ACL/共享内核/开放主机/上下游四种模式 |
| **对象类型** | Object Type | 本体中的实体类型定义，如「客户」、「订单」 |
| **对象实例** | Object Instance | 对象类型的具体实例，如具体的某位客户、某笔订单 |
| **属性** | Property | 描述对象特征的字段，如客户名称、订单金额 |
| **关系类型** | Link Type | 对象之间的语义关联，如「客户拥有订单」 |
| **聚合（聚合根）** | Aggregate / Aggregate Root | 以聚合根为核心的紧密关联对象集群，定义事务一致性边界 |
| **值对象** | Value Object | 无独立标识的概念性对象（如地址、金额），可嵌入实体 |
| **行为** | Behavior | 动力层唯一核心元语，对业务对象可执行的操作。受聚合根入口规则约束——挂在聚合根上的行为标记为「聚合入口行为」；行为定义中通过 events 数组声明式发布领域事件 |
| **行为类型** | Action Type | 行为的类型定义，如「审批订单」、「取消订单」 |
| **校验规则** | Validation Rule | 行为执行前后的业务规则，确保操作合规 |
| **事务边界** | Transaction Boundary | 界定聚合行为原子性的一致范围 |
| **语义层** | Semantic Layer | 本体架构的第一层，定义业务「名词」和「数据如何获取」。包含业务概念建模 + 数据源/数据映射/数据血缘 |
| **动力层** | Kinetic Layer | 本体架构的第二层，定义业务「动词」和「如何衡量」。包含行为定义 + 指标定义 |
| **事件层** | Event Layer | 本体架构的第三层，定义业务「信号」，实现聚合间松耦合通信 |
| **治理层** | Governance Layer | 本体架构的第四层，定义权限和治理（原动态层改名） |
| **领域事件** | Domain Event | 聚合内行为完成后的业务事实（如 OrderApproved），用过去式命名 |
| **集成事件** | Integration Event | 跨限界上下文/系统的外部事件，用于异步系统间通信 |
| **事件存储** | Event Store | 不可变的事件序列持久化，支持事件溯源和审计回放 |
| **事件总线** | Event Bus | 事件发布/订阅基础设施，支持内存队列和 Kafka/RabbitMQ |
| **MCP Server** | MCP Server | Model Context Protocol服务，AI与本体平台之间的协议适配层 |
| **MCP协议** | MCP | 大模型上下文协议，用于AI Agent与外部工具交互 |
| **幻觉** | Hallucination | 大模型生成看似合理但实际错误的内容 |
| **语义鸿沟** | Semantic Gap | AI系统无法理解业务语义的障碍 |
| **本体水合** | Ontology Hydration | 消费方根据平台记录的获取方式，从数据源获取实例数据并填充到本体对象的过程。平台仅提供获取方法元数据，不执行水合 |
| **数据源** | Data Source | 外部数据系统定义（ERP/CRM/MES等），包含连接配置、认证方式。本体平台不存储实例数据，只记录数据获取方法 |
| **数据获取方式** | Data Access Method | 属性级别的数据获取配置（仅元数据，不执行），支持三种方式：SQL直查、API调用（REST/OData/SOAP）、MCP工具调用 |
| **数据血缘** | Data Lineage | 数据字段的完整来源链路——从哪个数据源、哪个接口、经过什么转换而来 |
| **指标定义** | Metric Definition | 业务指标的形式化定义，包含聚合函数（SUM/COUNT/AVG等）、过滤条件、维度拆分和计算调度。是一种特殊的查询行为 |
| **指标维度** | Metric Dimension | 指标的分组维度，如按时间、按产品线、按区域拆分指标值 |
